## 1. 현재 프로젝트 한 줄

SomaMeet(소마밋)은 SW마에스트로 구성원들이 개인 일정 세부 내용을 공개하지 않고도 여러 사람의 공통 가능 시간을 빠르게 찾을 수 있도록 돕는 개인정보 보호 일정 조율 Agentic Workflow 프로젝트다.

서비스 관점에서는 여러 사람이 가능한 시간대만 제공하면 Agent가 공통 후보 시간을 계산하고 가장 적절한 회의 시간을 추천해주는 일정 조율 서비스다.

## 2. 핵심 결정사항

### D1. 주제

31조 프로젝트 주제는 `개인정보 보호가 되는 공유캘린더`로 둔다.

- 결정일: 2026-04-29(수)
- 이유: SW마에스트로에서는 팀 회의, 멘토링, 엑스퍼트 상담, 코칭처럼 여러 사람이 반복적으로 시간을 맞춰야 하지만 개인 일정 전체를 공유하기는 부담스럽다.
- 현재 한 줄 설명: 개인 일정의 세부 내용은 노출하지 않고, 가능한 시간대만 공유하여 여러 사람의 회의/멘토링/상담 일정을 빠르게 조율하는 공유캘린더 서비스
- 1차 대상 사용자: SW마에스트로 연수생, 팀원, 멘토, 엑스퍼트, 코치

### D2. MVP 방향

많은 기능보다 안정적인 단일 일정 조율 흐름에 집중한다.

MVP 핵심 흐름은 아래 순서다.

1. 회의 생성
2. 참여자별 가능 시간 입력
3. busy/free/preferred block 정규화
4. 공통 후보 시간 계산
5. 후보 3개 추천
6. 추천 이유와 복사용 공유 메시지 생성

최종 데모데이는 등수 경쟁보다 제품과 피칭을 보여주는 성격이므로, 기능 수보다 실제 소마 사람들이 이해하고 써볼 수 있는 안정성이 중요하다.

### D3. 개인정보 원칙

개인 일정 DB는 원본 캘린더 복제가 아니라 availability DB로 설계한다.

- 원본 일정 제목, 장소, 설명, 메모를 저장하지 않는다.
- 외부 캘린더 계정 접근 토큰을 저장하지 않는다.
- 회의 조율에 필요한 시작/종료 시간과 busy/free/preferred block만 저장한다.
- 계산은 deterministic scheduling engine이 맡고, Agent는 입력 정리, 추천 설명, 예외 처리, 공유 메시지 작성을 맡는다.

## 3. 문제와 사용자

### 해결하려는 문제

여러 사람이 참여하는 회의 일정을 잡을 때, 개인 일정은 보호하면서 공통 가능 시간을 빠르게 찾기 어렵다.

- 메신저 기반 조율은 응답이 흩어져 정리 시간이 오래 걸린다.
- 공유캘린더는 개인 일정의 제목, 장소, 내용이 드러날 수 있다.
- 사용자마다 Google Calendar, Apple Calendar, Outlook, 네이버 캘린더, 수기 일정표 등 서로 다른 도구를 쓴다.
- 일정 조율은 본질적인 개발/기획 작업은 아니지만 팀의 집중 시간을 계속 소모한다.

### 1차 사용자 시나리오

팀장이 팀원 3명과 멘토 1명의 1시간 멘토링 시간을 잡아야 한다. 각 참여자는 개인 일정 세부 내용 없이 가능한 시간 또는 바쁜 시간만 제공한다. SomaMeet은 입력을 공통 availability 형식으로 바꾸고, 공통 가능 시간을 계산한 뒤 후보 3개와 추천 이유를 보여준다. 팀장은 후보 중 하나를 고르고 Agent가 작성한 메시지를 단톡방에 공유한다.

## 4. Agent 설계

Agent는 일정 조율 담당자를 돕는 조율 보조자다.

Agent가 하는 일:

- 회의명, 소요 시간, 날짜 범위, 참여자, 필수 참석자 조건을 구조화한다.
- 직접 입력, `.ics`, CSV/텍스트 등 서로 다른 일정 입력을 공통 availability schema로 변환하는 흐름을 돕는다.
- 공통 가능 시간 계산 결과를 바탕으로 후보 3개를 추천한다.
- 추천 이유와 복사용 확정/재조율 메시지를 생성한다.
- 전원이 가능한 시간이 없으면 대안 후보와 추가 질문을 제안한다.

Agent가 하지 않는 일:

- 개인 일정 제목, 장소, 세부 내용 수집
- 외부 캘린더 계정 상시 접근
- 자동 일정 등록
- 자동 메시지 발송
- 사용자 승인 없는 최종 확정

핵심 원칙은 `계산은 규칙 기반, 설명은 Agent 기반`이다.

## 5. MVP 범위

반드시 구현할 것:

- 새 회의 생성
- 참여자별 가능 시간 직접 입력
- 개인 일정의 busy/free block을 공통 availability DB로 변환
- `.ics` 또는 CSV/텍스트 샘플 import 중 하나
- 30분 단위 공통 가능 시간 계산
- 후보 시간 3개 추천
- 추천 이유와 복사용 메시지 생성
- 로컬 실행 가능한 데모와 README

이번에는 하지 않을 것:

- Google Calendar, Apple Calendar 등 외부 캘린더 OAuth 실시간 연동
- 로그인/회원가입
- 모바일 앱
- 실시간 알림
- 자동 메시지 발송
- 자동 일정 등록
- 여러 회의 동시 최적화
- 반복 일정 최적화
- 원본 캘린더 이벤트 제목, 장소, 설명 저장

MVP 판단 기준:

> 소마 팀원 3명과 멘토 1명의 1시간 회의 시간을 안정적으로 잡을 수 있으면 성공이다.

## 6. 기술 구현 방향

2026-05-01 2차 오프라인 회의 고정본 기준으로, 기술 스택은 아직 완전히 확정하지 않았지만 아래 방향으로 임시 합의했다.

| 영역 | 현재 기준 |
| --- | --- |
| Frontend | React |
| Backend | Python, FastAPI |
| Deployment | AWS Lightsail 4GB |

기술 스택은 데모 구현 가능성과 팀원별 개발 분담 가능성을 기준으로 계속 검토한다.

아직 세부 확정이 필요한 영역:

- Agent/LLM workflow 구성 방식
- 데이터 검증과 API schema 관리 방식
- 일정 입력 방식: 직접 입력, `.ics`, CSV/text 중 데모에 포함할 범위
- availability 저장 방식: DB 또는 로컬 파일
- 공통 가능 시간 계산 엔진의 세부 구현
- 배포 후 사용자 테스트 범위

시스템 흐름:

```text
사용자/참여자
  -> React Frontend
  -> Calendar Source Adapter
      -> Manual Input / ICS Upload / CSV or Text Paste
      -> Privacy Filter
      -> Availability Normalizer
  -> FastAPI
  -> Scheduling/Agent Layer
      -> Time-slot Matching Engine
      -> Candidate Scorer
      -> LLM Message Generator
  -> Availability Storage
  -> 추천 후보 시간 + 공유 메시지 출력
```

공통 availability schema 예시:

```json
{
  "participant_id": "p1",
  "source_type": "ics",
  "timezone": "Asia/Seoul",
  "busy_blocks": [
    {
      "start": "2026-05-07T18:00:00+09:00",
      "end": "2026-05-07T19:00:00+09:00",
      "event_hash": "hash_only_no_title"
    }
  ]
}
```

## 9. AI 교육 맥락에서 보여줄 수 있는 요소

커리큘럼상 Prompt Engineering과 Agent Workflow가 주요 축이다. SomaMeet에서는 아래 요소를 작게라도 드러내면 좋다.

- Pydantic schema 기반 입력 구조화
- Tool Calling 또는 LangGraph 같은 workflow 도구로 `availability normalize -> 후보 계산 -> 메시지 생성` 흐름을 분리하는 방안 검토
- deterministic scheduling engine을 guardrail로 두어 LLM이 시간을 지어내지 못하게 하기
- 후보 없음, 입력 오류, 필수 참석자 부족 같은 예외 처리
- LLM-as-Judge 또는 rule-based checklist로 추천 결과 검증
- 개인정보 보호 원칙을 Safety/Security 항목으로 명시
