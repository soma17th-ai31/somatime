# 프로젝트 기획서 통합본

참여자: 박세종(팀장), 안수빈, 이중곤, 지상근, 홍지연

코치: 이원빈 코치님

## 1. 문제 정의 및 프로젝트 개요

### 1.1 프로젝트 한 줄 정의 [필수]

소마밋(SomaMeet)은 SW마에스트로 구성원들이 개인 일정 세부 내용을 공개하지 않고도 여러 사람의 공통 가능 시간을 빠르게 찾을 수 있도록 돕는 개인정보 보호 일정 조율 Agentic Workflow 프로젝트입니다.

### 1.2 서비스 한 줄 정의 [필수]

여러 사람이 각자 편한 방식으로 일정 정보를 입력하면, Agent가 개인 일정 내용은 숨기고 busy/free 정보만 활용해 모두가 가능한 회의 후보 시간을 추천해 주는 일정 조율 서비스입니다.

### 1.3 서비스 선정 배경 [권장]

SW마에스트로에서는 팀 회의, 멘토링, 엑스퍼트 상담, 코칭처럼 여러 사람이 시간을 맞춰야 하는 일이 반복적으로 발생합니다. 현재는 주로 메신저에서 "언제 가능하세요?"를 묻고 답하는 방식으로 조율하지만, 참여자가 늘어날수록 응답이 흩어지고 누가 언제 가능한지 다시 정리해야 합니다.

기존 도구도 한계가 있습니다. Google Calendar 같은 공유 캘린더는 자동 입력이 편리하지만 일정 제목, 장소, 설명까지 노출될 수 있습니다. When2Meet류 서비스는 익명성이 있지만 사용자가 시간 칸을 직접 클릭해야 해서 입력 마찰이 큽니다. 또한 사용자마다 Google Calendar, Apple Calendar, Outlook, 네이버 캘린더, 시간표 이미지, 수기 일정표 등 서로 다른 도구를 사용하기 때문에 특정 캘린더 하나에 의존하기 어렵습니다.

따라서 소마밋은 "개인 일정 내용은 가리고, 회의 조율에 필요한 가능 시간만 빠르게 모으는" 중간 지점을 목표로 합니다. 일정 조율에서 반복되는 수집, 정규화, 계산, 추천은 Agent가 처리하고, 최종 확정과 외부 발송은 사람이 승인하도록 분리합니다.

### 1.4 해결하려는 문제 [필수]

- 메신저 기반 일정 조율은 응답이 시간차로 흩어져 정리 시간이 오래 걸립니다.
- 참여자가 4명 이상만 되어도 가능한 시간 후보를 사람이 직접 비교하기 어렵고 실수가 생기기 쉽습니다.
- 공유 캘린더는 개인 일정 제목, 장소, 설명 같은 민감한 정보가 노출될 수 있습니다.
- 사용자마다 사용하는 캘린더나 시간표 형식이 달라 일정을 공통 형식으로 모으기 어렵습니다.
- 일정 조율은 본질적인 개발/기획 작업은 아니지만 팀의 집중 시간을 계속 소모합니다.

### 1.5 대상 사용자 [필수]

1차 대상 사용자는 SW마에스트로에서 3-7명 규모의 회의, 멘토링, 엑스퍼트 상담, 코칭 시간을 자주 조율해야 하는 소마인입니다.

구체적으로는 팀 회의 시간을 잡는 연수생 또는 팀장, 멘토링 일정을 맞춰야 하는 멘토, 여러 팀의 상담 요청을 받는 엑스퍼트, 팀 진행 상황을 확인해야 하는 코치가 사용할 수 있습니다. MVP에서는 특히 "팀장 또는 조율 담당자가 팀원 3-6명과 회의 시간을 잡는 상황"에 집중합니다.

### 1.6 핵심 가치 [필수]

핵심 가치는 프라이버시와 편의를 동시에 만족시키는 것입니다. 사용자는 개인 일정의 제목, 장소, 설명을 공개하지 않아도 되고, Agent는 회의 조율에 필요한 busy/free 시간 정보만 활용해 공통 가능 시간을 계산합니다.

결국 소마밋이 가장 잘해야 하는 일은 "여러 명에게 묻고 기다리는 시간"을 "한 번 입력하고 후보 3개를 받는 흐름"으로 줄이는 것입니다.

## 2. 사용자 및 Agent 설계

### 2.1 타깃 사용자 페르소나 [권장]

페르소나 A — 박조장

SW마에스트로 팀장 역할을 맡은 연수생입니다. 팀원 4-6명, 멘토, 코치와 회의 시간을 자주 맞춰야 합니다. 팀원들이 각자 다른 캘린더나 시간표를 쓰기 때문에 매번 메신저로 가능 시간을 물어보고 직접 비교합니다. 본인은 빠르게 회의 후보를 좁히고 싶지만, 팀원들의 개인 일정 세부 내용까지 알고 싶지는 않습니다.

페르소나 B — 김PM

소규모 프로젝트를 운영하는 PM형 사용자입니다. 주 1회 이상 5-7명 규모의 회의를 잡습니다. Google Calendar를 주로 쓰지만, 팀원은 Outlook, Apple Calendar, Notion, 종이 다이어리 등 서로 다른 방식으로 일정을 관리합니다. "언제 가능?"을 반복해서 묻는 시간을 줄이고 싶습니다.

페르소나 C — A멘토

여러 팀의 멘토링, 특강, 상담 요청을 동시에 받는 멘토입니다. 한 시간대에 여러 팀 요청이 겹칠 수 있고, 개인 업무 일정까지 함께 고려해야 합니다. 본업 일정의 내용은 공개하지 않으면서도 가능한 시간만 안전하게 공유하고 싶습니다.

### 2.2 Agent의 역할 [필수]

Agent는 "팀 회의 시간 조율 비서"로서 수집, 정규화, 계산, 추천을 담당합니다.

1. 사용자 등록 받기
   회의별로 사용자가 닉네임을 등록하면, Agent는 닉네임 단위로 일정 데이터를 관리합니다. MVP에서는 회원가입 없이 회의 링크와 닉네임 중심으로 흐름을 단순화합니다.

2. 개인 시간표 수집
   사용자는 편한 입력 방식을 선택합니다. 목표 기능은 Google Calendar 공유 링크, 캘린더 스크린샷 이미지, `.ics` 파일 입력입니다. MVP에서는 구현 안정성을 위해 `.ics` 파일, CSV/텍스트 입력, 직접 입력을 우선 지원하고, 스크린샷 인식과 실시간 OAuth 연동은 후순위 또는 확장 기능으로 둡니다.

3. 회의 조건 입력 받기
   Agent는 회의 제목, 후보 기간, 회의 길이, 참여 인원 수, 온라인/오프라인/상관없음 조건을 구조화합니다. 누락되거나 모호한 값이 있으면 추측하지 않고 한 번에 한 가지씩 짧게 되묻습니다.

4. 개인정보 필터링 및 정규화
   일정 제목, 장소, 설명은 저장하거나 추천 근거로 사용하지 않습니다. 시작 시간, 종료 시간, busy/free 여부, 온/오프라인 조건처럼 회의 조율에 필요한 정보만 공통 availability 형식으로 변환합니다.

5. 공통 가능 시간 계산
   닉네임별 busy/free 시간의 교집합을 구해 모두가 가능한 시간 슬롯을 찾습니다. 오프라인 회의인 경우 이동 시간을 고려해 슬롯 양옆에 30분 버퍼를 적용합니다.

6. 결과 출력
   Agent는 시간대별 가용 인원을 볼 수 있는 타임테이블과 조건을 가장 잘 만족하는 후보 시간 최대 3개를 제시합니다. 후보마다 추천 이유와 공유 메시지 초안을 함께 생성합니다.

### 2.3 Agent의 성격 및 톤앤매너 [권장]

Agent는 짧고 단정한 존댓말을 사용하는 실무형 일정 조율 비서처럼 동작합니다. 결과를 장황하게 설명하기보다 "타임테이블 + 후보 3개 + 짧은 추천 이유"의 정해진 형태로 응답합니다.

입력이 모호하면 임의로 추측해서 진행하지 않고 부족한 항목 1개만 짧게 되묻습니다. 예를 들어 회의 길이가 빠졌다면 "회의 길이는 몇 분으로 잡을까요?"처럼 확인합니다.

### 2.4 Agent의 자율성 범위 [필수]

Agent가 스스로 수행하는 일은 다음과 같습니다.

- 닉네임 등록과 회의별 일정 데이터 관리
- `.ics`, CSV/텍스트, 직접 입력 데이터의 공통 availability 변환
- 스크린샷 기능이 포함될 경우 이미지에서 일정 시간대 추출
- 회의 조건 기반 공통 가능 시간 계산
- 타임테이블 시각화 데이터 생성
- 후보 시간 최대 3개 추천
- 추천 이유와 회의 안내 메시지 초안 작성

사용자 확인이 필요한 지점은 다음과 같습니다.

- 스크린샷 또는 파일에서 일정을 추출했을 때 결과가 맞는지 확인
- 입력 항목 중 누락되거나 모호한 값이 있을 때 추가 응답
- 추천 후보 중 어떤 시간으로 확정할지 선택
- 회의 초대장, 알림, 메시지 발송 같은 외부 전송 행위 승인

Agent는 사용자 승인 없이 최종 회의 시간을 확정하거나, 외부 캘린더에 일정을 등록하거나, 메시지를 자동 발송하지 않습니다. 핵심 원칙은 "계산과 추천은 자동, 확정과 전송은 사람 승인 후"입니다.

## 3. 핵심 기능 및 사용자 흐름

### 3.1 주요 사용자 시나리오 [필수]

시나리오 1 — 멘토가 미뤄진 멘토 특강 시간을 잡는 경우

A멘토는 기존 일정 때문에 미뤄진 2시간짜리 오프라인 멘토 특강 시간을 다시 잡아야 합니다. A멘토는 소마밋에 닉네임을 등록하고 회의 조건을 입력합니다. 회의 제목은 "XXX 기술 특강", 후보 기간은 다음 주 평일, 길이는 120분, 참여 인원은 7명, 방식은 오프라인으로 설정합니다.

팀원들은 초대 링크에 접속해 닉네임을 등록하고, 본인이 편한 방식으로 일정을 입력합니다. 캘린더 파일이 있는 사람은 `.ics`를 업로드하고, 파일이 없는 사람은 CSV/텍스트 또는 직접 입력으로 busy/free 시간을 등록합니다. Agent는 닉네임별 일정의 교집합을 계산하고 오프라인 회의이므로 슬롯 양옆 30분 버퍼를 적용합니다. 이후 시간대별 가용 인원을 보여주는 타임테이블과 후보 시간 3개를 A멘토에게 제시합니다. A멘토는 그중 하나를 선택해 확정합니다.

시나리오 2 — 팀장이 캘린더 사용 여부가 다른 팀원과 조모임을 잡는 경우

B팀장은 팀원 4명과 다음 주 1시간 30분 온라인 조모임을 잡아야 합니다. 일부 팀원은 Google Calendar를 쓰지만, 한 명은 캘린더 앱을 거의 쓰지 않아 시간표를 이미지나 텍스트로만 공유할 수 있습니다.

B팀장은 소마밋에 회의 조건을 입력하고 초대 링크를 공유합니다. 캘린더를 쓰는 팀원은 `.ics` 파일이나 캘린더 데이터를 제출하고, 캘린더를 쓰지 않는 팀원은 직접 입력 또는 텍스트 형식으로 시간을 입력합니다. 스크린샷 인식 기능이 포함된 버전에서는 Agent가 추출 결과를 보여주고 "이렇게 맞나요?"라고 한 번 확인을 받습니다. 모든 응답이 모이면 Agent는 타임테이블과 후보 3개를 보여주고, B팀장은 최종 시간을 선택합니다.

### 3.2 핵심 기능 정의 [필수]

프로세스는 수집, 계산, 추천, 사용자 의사결정으로 나눕니다.

[수집]

1. 닉네임 단위 사용자 등록
   회의별로 닉네임을 등록하고, 해당 닉네임 단위로 일정 데이터를 관리합니다.

2. 회의 조건 입력
   회의 제목, 후보 기간, 회의 길이, 참여 인원 수, 온라인/오프라인/상관없음 조건을 입력합니다.

3. 개인 시간표 입력
   MVP에서는 `.ics` 파일, CSV/텍스트, 직접 입력을 우선 지원합니다. 확장 기능으로 Google Calendar 공유 링크 기반 free/busy 수집과 캘린더 스크린샷 Vision 추출을 검토합니다.

4. 추출 결과 확인
   파일 또는 이미지에서 자동 추출한 경우, 사용자에게 결과를 보여주고 한 번 검증받습니다.

[계산]

5. 개인정보 필터링
   일정 제목, 장소, 설명, 메모는 제거하고 시작 시간, 종료 시간, busy/free 여부만 사용합니다.

6. 공통 가능 시간 계산
   닉네임별 일정의 교집합을 계산해 모두가 가능한 시간 슬롯을 30분 단위로 도출합니다.

7. 이동 시간 버퍼
   오프라인 회의는 슬롯 양옆 30분 버퍼를 차감하고, 온라인 회의는 버퍼를 적용하지 않습니다. "상관없음"은 온라인 기준으로 계산하되, 추후 오프라인 후보도 함께 표시할 수 있습니다.

[추천]

8. 타임테이블 시각화
   입력된 날짜 범위에서 시간대별 가용 인원을 한눈에 볼 수 있는 표를 제공합니다.

9. 회의 후보 추천
   조건을 가장 잘 만족하는 슬롯을 최대 3개 추천하고, 각 후보의 추천 이유를 생성합니다.

10. 공유 메시지 초안
    사용자가 후보를 선택하면 단톡방이나 메신저에 붙여넣을 수 있는 회의 확정 메시지 초안을 생성합니다.

[사용자 의사결정]

11. 모호 입력 확인
    부족하거나 모호한 항목이 있으면 추측하지 않고 한 번에 1개만 짧게 질문합니다.

12. 최종 후보 선택
    Agent가 추천한 후보 중 어떤 시간으로 확정할지는 사용자가 직접 선택합니다.

13. 발송 승인
    회의 초대장, 알림 등 외부 전송 행위는 사용자가 "전송" 버튼을 누른 후에만 실행합니다.

### 3.3 사용자 관점 워크플로우 [필수]

[주최자 관점]

1. 소마밋에서 새 회의를 생성합니다.
2. 닉네임을 등록합니다.
3. 회의 조건을 입력합니다: 제목, 후보 기간, 길이, 인원, 온/오프라인 여부.
4. 초대 링크를 생성하고 메신저로 공유합니다.
5. 참여자 응답을 기다립니다.
6. Agent가 생성한 타임테이블과 후보 시간 3개를 확인합니다.
7. 후보 중 하나를 선택해 회의 시간을 확정합니다.
8. 메시지 초안을 확인하고, 필요한 경우 "전송" 또는 복사 버튼으로 공유합니다.

[참여자 관점]

1. 초대 링크에 접속합니다.
2. 닉네임을 등록합니다.
3. 일정 입력 방식을 선택합니다: `.ics` 업로드, CSV/텍스트 입력, 직접 입력, 추후 Google Calendar 링크 또는 스크린샷 업로드.
4. 자동 추출 결과가 있으면 화면에서 확인하고 수정합니다.
5. Agent가 누락·모호한 항목을 묻는 경우 답변합니다.
6. 일정을 제출합니다.
7. 주최자가 확정·공유하면 안내 메시지를 확인합니다.

### 3.4 시스템 관점 워크플로우 [권장]

```text
[회의 생성 - 주최자]
  회의 조건 구조화
  (제목 / 기간 / 길이 / 인원 / 온오프라인)
  -> 초대 링크 생성

[참여자 응답 수집]
  닉네임 등록
  -> Input Method Router
      -> ICS Parser
      -> CSV/Text Parser
      -> Manual Input Handler
      -> Screenshot Vision Extractor (확장)
      -> Google Calendar Adapter (확장)
  -> Privacy Filter
      일정 제목 / 장소 / 설명 제거
  -> Availability Normalizer
      닉네임 단위 busy/free block 통합
  -> Missing Field Checker
      모호 입력 1개만 되묻기

[공통 시간 계산]
  Intersection Engine
      닉네임별 일정 교집합, 30분 단위 계산
  -> Travel Buffer Applier
      오프라인 시 양옆 30분 차감

[결과 출력]
  Timetable Renderer
      시간대별 가용 인원 표시
  -> Candidate Ranker
      후보 최대 3개 산출
  -> Message Generator
      추천 이유와 공유 메시지 초안 생성

[사용자 의사결정]
  후보 선택
  -> 메시지 확인
  -> 사용자 승인 후 외부 공유
```

핵심 분리 원칙은 "계산은 deterministic scheduling engine, 설명과 메시지는 Agent"입니다. LLM이 시간을 지어내지 않도록 시간 후보 자체는 규칙 기반 계산 결과만 사용합니다.

## 4. 기술 구현 설계

### 4.1 기술 스택 [필수]

- Frontend: React
- Backend: Python, FastAPI
- Deployment: AWS Lightsail 4GB
- Data validation: Pydantic
- Calendar parsing: `icalendar`, CSV parser, `python-dateutil`, `zoneinfo`
- Storage: SQLite 또는 서버 임시 저장소
- Scheduling engine: Python deterministic time-slot calculation module
- LLM/Agent: OpenAI API 또는 교육 환경에서 사용 가능한 LLM API
- Vision/OCR: 스크린샷 인식은 MVP 이후 확장 또는 시간 여유가 있을 때 적용

### 4.2 시스템 아키텍처 [권장]

회의 생성자는 React 기반 프론트엔드에서 회의 조건을 입력합니다. 각 참여자는 같은 프론트엔드 화면에서 닉네임을 등록하고 자신의 일정 파일, CSV/텍스트, 직접 입력 값을 제출합니다.

프론트엔드는 입력값과 파일을 FastAPI 백엔드로 전달합니다. 백엔드는 입력 방식에 따라 Parser 또는 Adapter를 선택하고, 개인정보 필터를 통해 일정 제목, 장소, 설명을 제거합니다. 이후 참여자별 busy/free 시간 블록을 생성하고, 회의 조건에 맞는 공통 가능 시간을 계산합니다.

계산 결과는 타임테이블 데이터, 후보 시간 최대 3개, 추천 이유, 메시지 초안 형태로 프론트엔드에 반환됩니다. 프론트엔드는 사용자가 후보를 비교하고 최종 선택할 수 있도록 결과를 화면에 표시합니다.

전체 구조는 아래와 같습니다.

```text
React UI
  -> FastAPI
      -> Input Method Router
          -> ICS Parser
          -> CSV/Text Parser
          -> Manual Input Handler
          -> Screenshot Vision Extractor (확장)
          -> Google Calendar Adapter (확장)
      -> Privacy Filter
      -> Availability Normalizer
      -> Scheduling Engine
      -> Candidate Ranker
      -> LLM Message Generator
  -> SQLite / Temporary Storage
  -> React Result View
```

### 4.3 프롬프트 설계 전략 [권장]

Agent는 일정 추천 이유와 공유 메시지를 생성할 때 개인 일정 제목, 장소, 설명 같은 세부 내용을 언급하지 않습니다. 추천 근거는 busy/free 여부, 회의 길이, 날짜 범위, 참여 가능 인원, 온/오프라인 여부처럼 회의 조율에 필요한 정보로 제한합니다.

프롬프트 원칙은 다음과 같습니다.

- 시간 계산 자체는 LLM에 맡기지 않고 Scheduling Engine 결과만 사용합니다.
- LLM 입력에는 개인정보 필터링 후의 availability 데이터만 전달합니다.
- 후보 시간을 설명할 때는 "참여자 전원이 가능함", "요청한 회의 길이를 만족함", "입력한 날짜 범위 안에 있음", "오프라인 이동 버퍼를 반영함"처럼 검증 가능한 이유만 사용합니다.
- 불확실한 입력이나 누락된 참여자 정보가 있으면 확정적으로 말하지 않고 추가 확인이 필요한 조건을 안내합니다.
- 출력 형식은 "후보 3개 + 각 후보의 한 줄 이유 + 공유 메시지 초안"으로 고정합니다.

### 4.4 데이터 활용 및 기억 관리 [권장]

원본 캘린더 파일은 처리 후 삭제하는 것을 원칙으로 합니다. 서비스가 활용하는 데이터는 회의 조율에 필요한 최소 정보로 제한합니다.

저장하거나 활용하는 정보는 다음과 같습니다.

- 회의 메타데이터: 회의 제목, 후보 기간, 회의 길이, 참여 인원 수, 온/오프라인 여부
- 참여자 정보: 회의별 닉네임
- availability 데이터: 시작 시간, 종료 시간, busy/free 여부
- 후보 추천 결과: 후보 시간, 추천 이유, 메시지 초안

저장하지 않는 정보는 다음과 같습니다.

- 일정 제목, 장소, 설명, 메모
- 외부 캘린더 원본 파일의 장기 저장본
- 사용자 승인 없이 수집한 외부 계정 접근 정보
- LLM이 임의로 학습한 개인 선호 정보

MVP 단계에서는 사용자 계정이나 장기 메모리를 만들지 않습니다. 회의별 데이터만 사용하며, 필요한 경우 제출/데모가 끝난 뒤 데이터 삭제가 가능한 구조로 설계합니다.

### 4.5 제약사항 및 예외 처리 [필수]

- 캘린더 파일 형식이 지원되지 않거나 파일 내용이 깨져 있으면 파싱에 실패할 수 있습니다. 이 경우 직접 입력 또는 지원되는 CSV/텍스트 형식으로 다시 입력하도록 안내합니다.
- 스크린샷 인식은 Vision 모델 정확도와 이미지 품질에 영향을 받습니다. MVP에서는 후순위로 두고, 적용 시 추출 결과를 사용자에게 반드시 확인받습니다.
- 모든 참여자가 가능한 시간이 없으면 후보가 없음을 안내하고, 회의 길이를 줄이거나 날짜 범위를 넓히는 조건 완화를 제안합니다.
- 일부 참여자가 응답하지 않은 상태에서 후보 산출 요청이 들어오면, 응답한 사람 기준의 참고 후보임을 명시하고 미응답자를 표시합니다.
- 오프라인 회의는 이동 시간 버퍼 때문에 온라인보다 후보가 줄어들 수 있습니다. 이 경우 온라인 전환 또는 버퍼 완화를 대안으로 제시합니다.
- MVP에서는 로그인/회원가입, 자동 메시지 발송, 자동 일정 등록, 외부 캘린더 OAuth 실시간 연동, 복잡한 반복 일정, 다중 시간대 처리를 제외합니다.

### 5.4 기대 효과 [권장]

소마밋은 회의 시간 조율에 드는 메신저 핑퐁과 수동 비교 시간을 줄입니다. 팀장은 여러 사람의 가능 시간을 직접 모아 표로 정리하지 않아도 되고, 참여자는 개인 일정 세부 내용을 공개하지 않은 채 회의 가능 여부만 안전하게 제공할 수 있습니다.

또한 Agentic Workflow의 실용적인 예시를 보여줄 수 있습니다. 입력 정규화, 개인정보 필터링, 규칙 기반 시간 계산, LLM 기반 추천 이유와 메시지 생성, 사용자 승인 게이트가 하나의 흐름으로 연결됩니다. 이를 통해 단순 챗봇이 아니라 실제 작업을 단계별로 줄여주는 Agent 서비스를 데모할 수 있습니다.
