참여자 : 박세종(팀장), 안수빈, 이중곤, 지상근, 홍지연

이원빈 코치님

1. 문제 정의 및 프로젝트 개요

1.1 프로젝트 한 줄 정의 [필수]
프로젝트 전체를 한 문장으로 요약합니다.
"그래서 우리는 뭘 만드는 팀인가?"를 한 문장으로 설명할 수 있어야 합니다.

1.2 서비스 한 줄 정의 [필수]
사용자 관점에서 이 서비스가 무엇인지 한 문장으로 설명합니다.
프로젝트 설명과 별개로, 사용자가 봤을 때 서비스가 무엇인지 바로 이해할 수 있어야 합니다.

1.3 서비스 선정 배경 [권장]
왜 이 주제를 선택했는지 설명합니다.
문서의 맥락과 설득력을 높이는 항목입니다.

1.4 해결하려는 문제 [필수]
핵심으로 생각하는 문제와, 그것을 문제라고 생각한 이유를 함께 작성합니다.
"무엇을 만들까?"보다 "왜 이 기능이 필요한가?"를 기준으로 적습니다.

1.5 대상 사용자 [필수]
누가 이 Agent를 사용하는지 작성합니다.
대상 사용자가 정해져야 기능 수준, 난이도, 말투, 화면 흐름을 구체화할 수 있습니다.

1.6 핵심 가치 [필수]
이 프로젝트가 사용자에게 주는 가장 큰 이점을 작성합니다.
이 서비스가 딱 하나 가장 잘해야 하는 것이 무엇인지 정합니다.
2. 사용자 및 Agent 설계

2.1 타깃 사용자 페르소나 [권장]
연령대, 직업, 상황, 특징 등을 구체적으로 작성합니다.
처음부터 너무 자세하게 잡기보다, 실제 기능 설계에 연결되는 정도로 작성합니다.


2.2 Agent의 역할 [필수]
Agent가 정확히 어떤 일을 대신하는지, 어디까지 도와주는지 구체적으로 작성합니다.
단순 챗봇인지, 추천 도구인지, 실행 도구인지가 모호하지 않아야 합니다.


2.3 Agent의 성격 및 톤앤매너 [권장]
사용자가 Agent를 어떻게 느끼면 좋을지 작성합니다.
다만 가장 중요한 것은 말투보다 Agent가 수행하는 역할과 책임입니다.


2.4 Agent의 자율성 범위 [필수]
Agent가 어디까지 스스로 결정하고 실행할 수 있는지 작성합니다.
사람의 확인이 필요한 지점도 함께 적습니다.

3. 핵심 기능 및 사용자 흐름 

3.1 주요 사용자 시나리오 [필수]
사용자가 어떤 상황에서 이 서비스를 왜 쓰는지 구체적으로 작성합니다.
시나리오가 있어야 기능이 실제 사용 장면과 연결되고, 데모나 테스트도 쉬워집니다.

3.2 핵심 기능 정의 [필수]
무엇을 구현할지 정리합니다.
핵심 기능을 명확히 잡아야 범위가 계속 커지는 것을 막을 수 있습니다.

3.3 사용자 관점 워크플로우 [필수]
사용자가 처음부터 끝까지 어떤 순서로 서비스를 이용하는지 작성합니다.
입력과 출력의 흐름이 보이도록 단계별로 정리합니다.

3.4 시스템 관점 워크플로우 [권장]
시스템이 내부적으로 어떤 순서로 동작하는지 작성합니다.
처음부터 너무 복잡하게 그리기보다 구현에 필요한 수준으로 정리합니다.
4. 기술 구현 설계

4.1 기술 스택 [필수]
무엇으로 만들 것인지 작성합니다.
기술 스택이 정해져야 역할 분담과 구현 가능 범위를 판단할 수 있습니다.

4.2 시스템 아키텍처 [권장]
전체 구조를 이해할 수 있도록 작성합니다.
너무 복잡하게 잡기보다 데모 구현에 필요한 구조를 중심으로 적습니다.

4.3 프롬프트 설계 전략 [권장]
Agent의 역할, 입력, 출력 방식을 어떻게 통제할지 작성합니다.
없는 정보를 추측하지 않게 하는 원칙도 함께 적으면 좋습니다.


4.4 데이터 활용 및 기억 관리 [권장]
어떤 데이터를 활용하고, 어떤 정보를 저장하거나 저장하지 않을지 작성합니다.
서비스 특성에 따라 메모리와 개인정보 보호 범위를 함께 고려합니다.


4.5 제약사항 및 예외 처리 [필수]
무엇이 안 되는지, 어떤 상황에서 실패할 수 있는지, 이번 범위에서 제외하는 것이 무엇인지 작성합니다.
구현 과정에서 불필요한 욕심을 줄이고 완성도를 높이는 데 중요한 항목입니다.


5. 성과 평가 및 실행 계획

5.1 성공 지표(KPI) [권장]
무엇을 성공으로 볼지 작성합니다.
처음부터 정교한 지표가 아니어도, "이 정도면 성공이다"라는 기준은 필요합니다.


5.2 MVP 범위 [필수]
무엇을 반드시 만들고, 무엇은 이번에 하지 않을지 작성합니다.
MVP 범위가 없으면 기능이 계속 늘어나고 완성이 어려워집니다.


5.3 단계별 개발 로드맵 [필수]
어떤 순서로 만들지 작성합니다.
핵심 기능부터 차례대로 쌓는 흐름이 보이도록 정리합니다.


5.4 기대 효과 [권장]
이 서비스가 어떤 변화를 만들 수 있는지 작성합니다.
핵심 가치와 MVP가 이미 명확하다면 짧게 작성해도 됩니다.
