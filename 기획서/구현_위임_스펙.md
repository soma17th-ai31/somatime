# SomaMeet 구현 위임 스펙

> 이 문서의 목적
>
> - 다른 Claude Code(또는 다른 개발자)에게 SomaMeet MVP 구현을 위임할 때 사용하는 단일 기준 문서입니다.
> - 기획서(`구글독스_1차피드백_반영_통합본.md`)가 "왜/무엇"이라면, 이 문서는 "어떻게"입니다.
> - 위임받은 쪽은 이 문서를 그대로 따라 추측 없이 코드를 작성합니다.
> - 결정이 바뀌면 이 문서를 먼저 수정한 뒤 코드에 반영합니다.

---

## 0. 위임받은 쪽이 가장 먼저 할 일

1. 이 문서 전체를 읽는다.
2. 기획서 본문(`기획서/구글독스_1차피드백_반영_통합본.md`)을 읽는다.
3. 본 문서 9번 인수 기준 시나리오를 보고 "MVP 완성"의 정의를 머리에 넣는다.
4. 1주차 로드맵(기획서 5.3)을 따라 작업 시작.
5. 의사결정이 필요한 모호한 지점이 발견되면 추측하지 말고 PR 코멘트나 이슈로 질문한다.

## 4. API 엔드포인트

전부 `/api` 프리픽스. 응답은 모두 JSON. 에러는 `{ "error_code": "...", "message": "...", "suggestion": "..." }` 형식.

| # | Method | Path | 권한 | 설명 |
|---|---|---|---|---|
| 1 | POST | `/api/meetings` | 누구나 | 회의 생성 |
| 2 | GET | `/api/meetings/{slug}` | URL만 | 회의 정보 조회 |
| 3 | POST | `/api/meetings/{slug}/participants` | 누구나 | 닉네임 등록 + 쿠키 발급 |
| 4 | POST | `/api/meetings/{slug}/availability/manual` | 본인 쿠키 | 직접 입력 |
| 5 | POST | `/api/meetings/{slug}/availability/ics` | 본인 쿠키 | ICS 업로드 (multipart) |
| 6 | GET | `/api/meetings/{slug}/availability/google/oauth-url` | 본인 쿠키 | OAuth URL 발급 |
| 7 | GET | `/api/auth/google/callback` | OAuth state | 콜백 → free/busy 수집 |
| 8 | GET | `/api/meetings/{slug}/timetable` | URL만 | 시간대별 가용 인원 |
| 9 | POST | `/api/meetings/{slug}/calculate` | URL만 | 후보 3개 계산 |
| 10 | POST | `/api/meetings/{slug}/confirm` | **organizer 토큰** | 확정 + 메시지 초안 |

## 5. 접근 제어 모델 (옵션 A)

### URL 구조
```
주최자 전용:  /m/{slug}?org={organizer_token}
참여자 공유용: /m/{slug}
```

### 토큰 발급
- `slug`: 8자 base62 (예: `aB3kF9xQ`)
- `organizer_token`: `secrets.token_urlsafe(24)` → 32자 이상
- `participant_token`: 참여자 등록 시 발급, **HttpOnly 쿠키** `somameet_pt_{slug}` 로 저장

### 권한 매트릭스

| Endpoint | 필요 권한 | 검증 방법 |
|---|---|---|
| `POST /api/meetings` | 누구나 | - |
| `GET /api/meetings/{slug}` | URL만 | - |
| `POST /participants` | 누구나 | 응답 시 쿠키 set |
| `POST /availability/*` | 본인 participant | 쿠키 |
| `GET /timetable` | URL만 | - |
| `POST /calculate` | URL만 | - |
| `POST /confirm` | **주최자만** | `X-Organizer-Token` 헤더 |
| `DELETE /api/meetings/{slug}` | **주최자만** | 동일 |

### 프론트 UX 규칙
- 회의 생성 직후 화면에 두 URL을 분리해서 표시:
  ```
  [내 관리용 링크 — 공유 금지]  /m/aB3kF9xQ?org=...   [복사]
  [팀원에게 공유할 링크]         /m/aB3kF9xQ           [복사]
  ```
- 주최자 URL로 들어오면 "확정" 버튼 노출
- 참여자 URL로 들어오면 "확정" 버튼 미노출 (하드 가드는 백엔드)

## 6. 알고리즘 규칙

| 항목 | 결정 |
|---|---|
| 슬롯 단위 | **30분** (busy_blocks 30분 경계로 내림 정규화) |
| 검색 시간대 | 주최자 입력 (기본 09:00-22:00) |
| 주말 포함 | 주최자 토글 (기본 OFF) |
| 오프라인 버퍼 | 후보 `[start-30min, end+30min]` 전 구간 free일 때만 인정 |
| 후보 랭킹 | (1) 가용 인원 (2) 시간대 분산 — 직전 후보와 2h+ 떨어진 것 우선 (3) 빠른 날짜 |
| 타임존 | Asia/Seoul 고정 (모든 datetime KST 저장) |
| `any` 처리 | 온라인 기준 (버퍼 미적용) |
| 후보 0개 폴백 | 1명 빠진 후보 → 그것도 0이면 조건 완화 메시지 |
| 데이터 충돌 | 같은 닉네임 재입력 시 last-write-wins (트랜잭션) |

## 7. LLM 사용 경계

| 작업 | LLM 사용 여부 |
|---|---|
| 공통 가능 시간 계산 | ❌ deterministic 코드 |
| 입력 정규화(ICS 파싱, 직접 입력 검증) | ❌ deterministic 코드 |
| 누락 항목 검증 ("회의 길이 입력 안 됨") | ❌ Pydantic + 단순 체크 |
| 추천 이유 문자열 생성 | ✅ LLM (1회 호출, 후보 3개 묶어서) |
| 안내 메시지 초안 생성 | ✅ LLM (확정 시 1회) |
| 후보 0개일 때 완화 제안 문구 | ✅ LLM 또는 템플릿 (구현자 선택) |

LLM 호출은 회의 1건당 최대 3회 정도. 비용 최소화.

**Privacy 가드:** LLM에 보내는 프롬프트에 일정 제목/설명/위치 절대 포함 금지. 슬롯 시간, 가용 인원, 회의 메타데이터만 전달.

### 7.1 LLM 어댑터 패턴 (provider 교체 가능)

LLM 호출은 단일 인터페이스 뒤에 숨기고, `LLM_PROVIDER` 환경 변수로 런타임 선택. 키 발급 전에는 `template` 으로 시작해서 흐름부터 검증.

```python
# backend/app/services/llm/__init__.py
from app.core.config import settings

def get_llm_adapter():
    provider = settings.LLM_PROVIDER
    if provider == "gemini":
        from .gemini import GeminiAdapter
        return GeminiAdapter()
    elif provider == "anthropic":
        from .anthropic import AnthropicAdapter
        return AnthropicAdapter()
    elif provider == "openai":
        from .openai import OpenAIAdapter
        return OpenAIAdapter()
    elif provider == "template":
        from .template import TemplateAdapter
        return TemplateAdapter()
    else:
        raise ValueError(f"Unknown LLM_PROVIDER: {provider}")
```

```python
# backend/app/services/llm/base.py
from abc import ABC, abstractmethod

class LLMAdapter(ABC):
    @abstractmethod
    def generate_recommendation_reasons(
        self,
        candidates: list[Candidate],
        meeting: Meeting,
    ) -> list[str]:
        """후보별 추천 이유 문자열 반환. 길이 = len(candidates)"""

    @abstractmethod
    def generate_share_message(
        self,
        meeting: Meeting,
        confirmed_slot: Slot,
        nicknames: list[str],
    ) -> str:
        """공유 메시지 초안 반환"""
```

```python
# backend/app/services/llm/template.py — 키 없을 때 폴백
class TemplateAdapter(LLMAdapter):
    def generate_recommendation_reasons(self, candidates, meeting):
        return [
            f"참여자 {c.available_count}명 가능, "
            f"{['월','화','수','목','금','토','일'][c.start.weekday()]}요일 "
            f"{c.start.strftime('%H:%M')}"
            for c in candidates
        ]

    def generate_share_message(self, meeting, slot, nicknames):
        return (
            f"'{meeting.title}' 일정 확정 안내드립니다.\n"
            f"일시: {slot.start:%Y-%m-%d %H:%M} - {slot.end:%H:%M}\n"
            f"참여자: {', '.join(nicknames)}"
        )
```

**규칙:**
- 모든 LLM 호출은 `get_llm_adapter()` 를 통해서만 (직접 SDK 호출 금지)
- 새 provider 추가는 `LLMAdapter` 상속 클래스 1개 만들고 factory에 등록만 하면 끝
- `template` 은 키 발급 전 개발/테스트 기본값. 인수 시나리오 S11(privacy 검증)도 `template` 으로는 자동 통과 — 키 받은 뒤 실제 provider로 다시 검증해야 함.

## 9. 인수 기준 시나리오 (MVP 완성 정의)

위임받은 쪽이 `backend/tests/acceptance/test_*.py` 에 그대로 구현. **전부 통과해야 MVP 완성.**

### S1. Happy path — 회의 생성부터 확정까지
- **Given** 주최자 A, 참여자 B/C/D
- **When** A가 회의 생성(60분, online, 평일) → B/C/D가 manual/ics/manual로 입력 → A가 calculate → 첫 후보 confirm
- **Then**
  - 회의 생성 응답 status 201, `slug` 길이 8, `organizer_token` 길이 32+
  - calculate 응답 `candidates` 길이 ≤ 3
  - 모든 후보가 3명 교집합 안
  - confirm 응답에 `confirmed_slot`과 `share_message_draft`
  - `share_message_draft` 에 일정 제목/장소/설명 단어 없음

### S2. 오프라인 버퍼 검증
- **Given** offline 회의, 길이 60분, 1명이 12:00-13:30 busy + 14:30-16:00 busy
- **Then** 13:30-14:30 슬롯은 후보 **제외** (양옆 30분 위반)
- **Then** 같은 입력으로 online이면 13:30-14:30 후보 **포함**

### S3. 후보 0개 → "1명 빠진 후보" 폴백
- **Given** 4명 중 1명이 후보 기간 전부 busy
- **Then** 응답 후보에 `missing_participants: ["닉네임"]` 포함, `note: "OO님 제외 가능"`

### S4. 폴백조차 0개 → 조건 완화 제안
- **Given** 모든 참여자가 모든 시간대 busy
- **Then** `candidates: []`, `suggestion`에 "회의 길이 줄이거나 날짜 범위 넓히세요" 메시지, status **200**

### S5. 권한 — 주최자 토큰
- `organizer_token` 헤더 없이 confirm → **403**
- 잘못된 토큰 → **403**
- `slug`만으로 GET 회의 → **200**

### S6. 권한 — 참여자 격리
- B의 쿠키로 C의 응답 수정 시도 → **403**

### S7. 같은 닉네임 재입력 (last-write-wins)
- B가 manual 입력 후 ICS 업로드 → 기존 manual의 busy_blocks 모두 삭제, 새 ICS만 남음
- 트랜잭션 검증

### S8. ICS 파싱 실패
- 빈 파일/깨진 파일 업로드 → **400**, `error_code: "ics_parse_failed"`, `suggestion` 포함
- DB에 부분 데이터 저장 안 됨

### S9. Google OAuth — free/busy 만 요청
- OAuth URL의 `scope` 파라미터가 `calendar.freebusy` 만 포함
- `calendar.readonly`, `calendar.events` 미포함
- 콜백 후 DB의 busy_blocks에 시간만 있음 (제목 컬럼 자체가 스키마에 없음)

### S10. 타임테이블 응답 형식
- 30분 단위 슬롯 배열, 각 슬롯에 `start`, `end`, `available_count`, `available_nicknames: []`
- 닉네임만 포함 (이메일/실명 없음)

### S11. LLM 출력 privacy 검증
- 참여자가 ICS에 "병원 진료", "데이트" 이벤트 포함
- `share_message_draft` 에 "병원", "진료", "데이트" 단어 **없음**
- 추천 이유는 일반화된 문장만

### E1. 브라우저 골든패스 (Playwright)
1. `/` 접속 → 회의 생성 → 결과 화면에 두 URL 표시
2. 공유 링크를 시크릿 창에서 열기 → "확정" 버튼 미노출
3. 닉네임 등록 → manual 입력 → 제출
4. 주최자 창으로 돌아가 calculate → 후보 3개 표시
5. 첫 후보 선택 → 메시지 초안 모달 → 복사 버튼
