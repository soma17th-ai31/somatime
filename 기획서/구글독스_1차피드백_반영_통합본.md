# 프로젝트 기획서

참여자: 박세종(팀장), 안수빈, 이중곤, 지상근, 홍지연

코치: 이원빈 코치님

## 1. 문제 정의 및 프로젝트 개요 (박세종)

### 1.1 프로젝트 한 줄 정의 [필수]

소마밋(SomaMeet)은 SW마에스트로 구성원들이 개인 일정 세부 내용을 공개하지 않고도 여러 사람의 공통 가능 시간을 빠르게 찾을 수 있도록 돕는 개인정보 보호 일정 조율 Agentic Workflow 프로젝트입니다.

### 1.2 서비스 한 줄 정의 [필수]

여러 사람이 각자 편한 방식으로 일정 정보를 입력하면, Agent가 개인 일정 내용은 숨기고 busy/free 정보만 활용해 모두가 가능한 회의 후보 시간을 추천해 주는 일정 조율 서비스입니다.

### 1.3 서비스 선정 배경 [권장]

SW마에스트로에서는 팀 회의, 멘토링, 엑스퍼트 상담, 코칭처럼 여러 사람이 시간을 맞춰야 하는 일이 반복적으로 발생합니다. 현재는 주로 메신저에서 "언제 가능하세요?"를 묻고 답하는 방식으로 조율하지만, 참여자가 늘어날수록 응답이 흩어지고 누가 언제 가능한지 다시 정리해야 합니다.

기존 도구도 한계가 있습니다. Google Calendar 같은 공유 캘린더는 자동 입력이 편리하지만 일정 제목, 장소, 설명까지 노출될 수 있습니다. When2Meet류 서비스는 익명성이 있지만 사용자가 시간 칸을 직접 클릭해야 해서 입력 마찰이 큽니다. 또한 사용자마다 Google Calendar, Apple Calendar, Outlook, 네이버 캘린더, 시간표 이미지, 수기 일정표 등 서로 다른 도구를 사용하기 때문에 특정 캘린더 하나에 의존하기 어렵습니다.

따라서 소마밋은 "개인 일정 내용은 가리고, 회의 조율에 필요한 가능 시간만 빠르게 모으는" 중간 지점을 목표로 합니다. 일정 조율에서 반복되는 수집, 정규화, 계산, 추천은 Agent가 처리하고, 최종 확정과 외부 공유는 사람이 확인하도록 분리합니다.

### 1.4 해결하려는 문제 [필수]

- 메신저 기반 일정 조율은 응답이 시간차로 흩어져 정리 시간이 오래 걸립니다.
- 참여자가 4명 이상만 되어도 가능한 시간 후보를 사람이 직접 비교하기 어렵고 실수가 생기기 쉽습니다.
- 공유 캘린더는 개인 일정 제목, 장소, 설명 같은 민감한 정보가 노출될 수 있습니다.
- 사용자마다 사용하는 캘린더나 시간표 형식이 달라 일정을 공통 형식으로 모으기 어렵습니다.
- 일정 조율은 본질적인 개발/기획 작업은 아니지만 팀의 집중 시간을 계속 소모합니다.

### 1.5 대상 사용자 [필수]

1차 대상 사용자는 SW마에스트로에서 3-7명 규모의 회의, 멘토링, 엑스퍼트 상담, 코칭 시간을 자주 조율해야 하는 소마인입니다.

구체적으로는 팀 회의 시간을 잡는 연수생 또는 팀장, 멘토링 일정을 맞춰야 하는 멘토, 여러 팀의 상담 요청을 받는 엑스퍼트, 팀 진행 상황을 확인해야 하는 코치가 사용할 수 있습니다. MVP에서는 특히 "팀장 또는 조율 담당자가 팀원 3-6명과 회의 시간을 잡는 상황"에 집중합니다.

### 1.6 핵심 가치 [필수]

핵심 가치는 프라이버시와 편의를 동시에 만족시키는 것입니다. 사용자는 개인 일정의 제목, 장소, 설명을 공개하지 않아도 되고, Agent는 회의 조율에 필요한 busy/free 시간 정보만 활용해 공통 가능 시간을 계산합니다.

결국 소마밋이 가장 잘해야 하는 일은 "여러 명에게 묻고 기다리는 시간"을 "한 번 입력하고 후보 3개를 받는 흐름"으로 줄이는 것입니다.

## 2. 사용자 및 Agent 설계 (안수빈)

### 2.1 타깃 사용자 페르소나 [권장]

페르소나 A — 김PM (29세, SW마에스트로 팀장형 연수생)

주 1회 이상 5-7명 규모의 팀 회의나 멘토링 일정을 직접 잡습니다. 팀원들이 각자 다른 캘린더(구글, 노션, 종이 다이어리)를 쓰는 탓에 "언제 가능?"을 메신저로 매번 묻고 있습니다. 본인은 Google Calendar를 메인으로 씁니다.

페르소나 B — 박조장 (22세, SW마에스트로 팀 프로젝트 조장)

팀원이 4-5명입니다. 팀원마다 멘토링, 특강, 개인 일정이 제각각이라 회의 시간 잡는 데 며칠씩 걸립니다. 일부 팀원은 캘린더 앱 자체를 잘 안 써서 직접 입력이나 캘린더 내보내기 파일로 일정을 공유받는 경우가 많습니다.

### 2.2 Agent의 역할 [필수]

Agent는 "팀 회의 시간 조율 비서"로서 다음 일을 수행합니다.

1. 사용자 등록 받기
   각 팀원이 닉네임을 설정하면 그 닉네임 단위로 일정 데이터를 관리합니다.

2. 개인 시간표 수집
   MVP에서는 다음 입력 방식 중 사용자가 편한 것을 선택하면, Agent가 그것을 정규화된 일정 데이터로 변환합니다.
   - Google Calendar 공유 링크: 캘린더 API로 free/busy 정보만 수집
   - 캘린더 내보내기 파일(.ics): 파일을 파싱해 일정 추출
   - 직접 입력: 사용자가 가능한 시간 또는 바쁜 시간을 직접 입력

   캘린더 스크린샷 이미지에서 일정 시간대를 추출하는 Vision 기능은 확장 기능으로 둡니다.

3. 회의 조건 입력 받기
   회의 제목, 회의 날짜(또는 후보 기간), 회의 시간(길이), 참여 인원 수, 온라인/오프라인/상관없음 조건을 받습니다.

4. 공통 가능 시간 계산
   닉네임별 일정의 교집합을 구해 모두가 가능한 시간 슬롯을 찾습니다. 온라인/오프라인 조건에 따라 이동 시간 가정을 다르게 적용합니다. 오프라인은 슬롯 양옆 30분 버퍼 등을 적용합니다.

5. 결과 출력
   - 타임테이블 시각화: 입력된 날짜의 시간대별 가용 인원을 한눈에 볼 수 있는 표를 제공합니다.
   - 회의 시간 후보 추천: 조건을 가장 잘 만족하는 시간 슬롯을 최대 3개 추천합니다.

### 2.3 Agent의 성격 및 톤앤매너 [권장]

말투는 짧고 단정한 존댓말 비서 스타일입니다. 결과를 길게 설명하기보다 "타임테이블 + 후보 3개"라는 정해진 형태로 응답합니다. 입력이 모호하면 추측해서 실행하지 않고 부족한 항목 1개만 짧게 되묻습니다. 예를 들어 "회의 길이는 몇 분으로 잡을까요?"처럼 확인합니다.

다만 가장 중요한 것은 말투가 아니라, 위에서 정의한 "수집 → 계산 → 추천"이라는 책임을 정확히 수행하는 것입니다.

### 2.4 Agent의 자율성 범위 [필수]

Agent가 스스로 수행하는 일은 다음과 같습니다.

- 닉네임 등록과 일정 데이터 저장
- Google Calendar free/busy 정보, .ics 파일, 직접 입력에서 일정 데이터 정규화
- 확장 기능으로 스크린샷에서 일정 자동 추출
- 입력된 조건(제목, 날짜, 길이, 인원, 온/오프라인) 기반 공통 가능 시간 계산
- 타임테이블 시각화 및 후보 시간 최대 3개 추천

사용자 확인이 필요한 지점은 다음과 같습니다.

- Google Calendar, .ics, 직접 입력 결과가 맞는지 사용자에게 한 번 확인 요청
- 입력 항목 중 누락되거나 모호한 값이 있을 때 추측 없이 되묻기
- 추천된 후보 중 어떤 시간으로 확정할지 사용자 직접 선택
- 회의 안내 메시지는 초안으로만 생성하고, 사용자는 이를 직접 복사해 외부 채널에 공유

## 3. 핵심 기능 및 사용자 흐름 (지상근)

### 3.1 주요 사용자 시나리오 [필수]

페르소나별 대표 시나리오는 다음 두 가지입니다.

시나리오 1 — A멘토가 일정 때문에 미뤄진 멘토 특강 시간을 잡는 경우

A멘토는 기존 일정 때문에 미뤄진 2시간짜리 오프라인 멘토 특강 시간을 잡아야 합니다. 소마밋에 본인 닉네임을 등록하고 회의 조건(제목 "XXX 기술", 후보 기간: 다음 주 평일, 길이: 120분, 인원: 7명, 오프라인)을 입력한 뒤 팀원에게 초대 링크를 공유합니다.

팀원들은 본인이 편한 방식인 Google Calendar 공유 링크, .ics 파일, 직접 입력 중 하나로 일정을 등록합니다. Agent는 닉네임별 일정의 교집합을 계산하고 오프라인 회의이므로 슬롯 양옆 30분 버퍼를 적용한 뒤, 시간대별 가용 인원을 보여주는 타임테이블과 후보 시간 3개를 A멘토에게 제시합니다. A멘토는 그중 하나를 선택해 확정합니다.

시나리오 2 — B팀 조장이 캘린더 사용 여부가 다른 조원과 조모임을 잡는 경우

B팀 조장은 조원 4명과 다음 주 1시간 30분 온라인 조모임을 잡아야 합니다. 조원 중 일부는 Google Calendar를 쓰지만 한 명은 캘린더 앱을 거의 쓰지 않아 가능한 시간을 직접 입력해야 합니다.

B팀 조장은 소마밋에 회의 조건(온라인)을 입력하고 초대 링크를 공유합니다. 캘린더를 쓰는 조원은 .ics 또는 Google Calendar로, 안 쓰는 조원은 직접 입력으로 일정을 제출합니다. Agent는 자동 수집 또는 입력된 결과를 보여주고 "이렇게 맞나요?"라고 한 번 확인을 받습니다. 입력 중 누락 항목(예: 회의 길이)이 있으면 추측 없이 "회의 길이는 몇 분으로 잡을까요?"처럼 1개만 짧게 되묻습니다. 모든 응답이 모이면 타임테이블과 후보 3개를 보여줍니다.

### 3.2 핵심 기능 정의 [필수]

프로세스는 수집, 계산, 추천으로 나눕니다.

[수집]

1. 닉네임 단위 사용자 등록
   회의별로 닉네임을 등록하면 그 단위로 일정 데이터를 관리합니다.

2. 회의 조건 입력
   회의 제목, 회의 날짜(또는 후보 기간), 회의 길이, 참여 인원 수, 온라인/오프라인/상관없음 5가지를 입력합니다.

3. 개인 시간표 입력 3종
   사용자가 편한 방식을 선택합니다.
   - Google Calendar 공유 링크: 캘린더 API로 free/busy 정보만 수집
   - .ics 내보내기 파일: 파일 파싱으로 일정 추출
   - 직접 입력: 가능한 시간 또는 바쁜 시간 직접 입력

   캘린더 스크린샷 이미지를 Vision으로 추출하는 기능은 MVP 이후 확장 기능으로 둡니다.

4. 추출 결과 확인
   Google Calendar free/busy 수집 결과, .ics 파싱 결과, 직접 입력 결과를 사용자에게 보여주고 한 번 검증합니다.

[계산]

5. 공통 가능 시간 계산
   닉네임별 일정의 교집합으로 모두가 가능한 시간 슬롯을 도출합니다.

6. 이동 시간 버퍼
   오프라인 회의는 슬롯 양옆 30분 버퍼를 차감하고, 온라인 회의는 버퍼를 적용하지 않습니다. "상관없음"은 온라인 기준으로 적용합니다.

[추천]

7. 타임테이블 시각화
   입력된 날짜 범위에서 시간대별 가용 인원을 한눈에 보여주는 표를 제공합니다.

8. 회의 후보 추천
   조건을 가장 잘 만족하는 슬롯을 최대 3개 추천합니다.

[사용자 의사결정]

9. 모호 입력 1개만 되묻기
   부족하거나 모호한 항목이 있으면 추측하지 않고 한 번에 1개만 짧게 질문합니다.

10. 최종 후보 선택
    Agent가 추천한 후보 중 어떤 시간으로 확정할지는 사용자가 직접 선택합니다.

11. 메시지 초안 확인 및 복사
    회의 안내 메시지는 초안으로 생성하고, 사용자가 확인한 뒤 직접 복사해 외부 채널에 공유합니다. 이메일, 메신저, 캘린더 초대장 같은 실제 외부 발송 자동화는 이번 범위에서 제외합니다.

### 3.3 사용자 관점 워크플로우 [필수]

주최자 관점 워크플로우는 다음과 같습니다.

1. 닉네임 등록
2. 회의 조건 입력: 제목, 후보 기간, 길이, 인원, 온/오프라인
3. 초대 링크 생성 후 메신저로 공유
4. 참여자 응답 대기
5. Agent가 타임테이블과 후보 시간 3개 제시
6. 후보 중 하나 선택해 확정
7. 안내 메시지 초안을 확인하고 복사해 직접 공유

참여자 관점 워크플로우는 다음과 같습니다.

1. 초대 링크 접속 후 닉네임 등록
2. 일정 입력 방식 선택: Google Calendar 공유 링크, .ics 파일 업로드, 직접 입력
3. Google Calendar, .ics, 직접 입력 결과를 화면에서 확인하고 맞으면 다음 단계로 진행
4. 누락되거나 모호한 항목이 있으면 Agent가 1개만 짧게 되묻고 사용자가 응답
5. 제출
6. 주최자가 확정하고 메시지 초안을 공유하면 안내 메시지 확인

### 3.4 시스템 관점 워크플로우 [권장]

## 4. 기술 구현 설계 (이중곤)

### 4.1 기술 스택 [필수]

- Frontend: React
- Backend: Python, FastAPI
- Deployment: AWS Lightsail 4GB
- Google Calendar API: free/busy 조회
- OAuth 2.0: Google Calendar 권한 승인
- Data validation: Pydantic
- Calendar parsing: `icalendar` 또는 CSV parser
- Storage: SQLite 또는 서버 임시 저장소

### 4.2 시스템 아키텍처 [권장]

- 회의 생성자는 React 기반 프론트엔드 화면에서 회의 조건을 입력합니다.
- 각 참여자는 같은 프론트엔드 화면에서 자신의 이름과 Google Calendar 공유 링크, 캘린더 파일, 직접 입력 값을 추가하고, 프론트엔드는 입력값과 파일을 FastAPI 백엔드로 전달합니다.
- 백엔드는 Google Calendar API의 free/busy 정보 또는 캘린더 파일을 처리한 뒤 일정 제목, 장소, 설명을 제거합니다. 이후 참여자별 busy/free 시간 블록을 생성하고, 회의 조건에 맞는 공통 가능 시간을 계산합니다.
- 계산 결과인 모든 참여자의 busy/free 시간 블록이 등록된 타임 테이블, 후보 시간 3개, 추천 이유를 프론트엔드에 반환합니다.
- 프론트엔드는 사용자가 후보 시간을 비교하고 최종 선택할 수 있도록 결과를 화면에 보여줍니다.

### 4.3 프롬프트 설계 전략 [권장]

Agent는 일정 추천 이유를 생성할 때 입력된 일정의 제목, 장소, 설명 같은 세부 내용을 언급하지 않습니다. 추천 근거는 busy/free 여부, 회의 길이, 날짜 범위, 온/오프라인 여부처럼 회의 조율에 필요한 정보로 제한합니다.

후보 시간을 설명할 때는 "참여자 전원이 가능함", "요청한 회의 길이를 만족함", "입력한 날짜 범위 안에 있음"처럼 검증 가능한 이유만 사용합니다.

불확실한 입력이 있거나 일부 참여자의 일정 정보가 부족한 경우에는, 사용자가 추가로 확인해야 할 조건을 함께 안내합니다.

### 4.4 데이터 활용 및 기억 관리 [권장]

원본 캘린더 파일은 처리 후 삭제하는 것을 원칙으로 합니다. 서비스가 활용하는 데이터는 회의 조율에 필요한 최소 정보로 제한합니다.

일정 제목, 장소, 설명은 저장하지 않고, 시작 시간, 종료 시간, busy/free 여부만 공통 형식으로 변환해 사용합니다.

Google Calendar API를 사용하는 경우에도 free/busy 조회 권한만 요청합니다. 이벤트 제목, 장소, 설명을 읽는 권한은 요청하지 않으며, OAuth 토큰은 MVP 단계에서 장기 저장하지 않고 세션 또는 서버 임시 저장소에 필요한 기간만 보관합니다.

### 4.5 제약사항 및 예외 처리 [필수]

- Google Calendar 권한 승인이 실패하거나 캘린더 파일 형식이 지원되지 않는 경우 파싱에 실패할 수 있습니다. 이 경우 사용자가 직접 busy/free 시간대를 입력하거나 지원되는 샘플 형식으로 다시 업로드하도록 안내합니다.
- 모든 참여자가 가능한 시간이 없을 경우에는 조건을 만족하는 후보가 없음을 안내합니다. 이때 회의 길이를 줄이거나 날짜 범위를 넓히는 방식으로 조건 완화를 제안합니다.
- 캘린더 스크린샷 Vision 인식은 MVP에 포함하지 않습니다. 이후 확장 시 인식 결과를 사용자에게 반드시 확인받습니다.

### 5.2 MVP 범위 [필수]

MVP 범위는 다음과 같습니다.

- 닉네임 등록 및 초대 링크 생성
- .ics 파일 파싱 및 busy/free 추출
- Google Calendar API를 통한 free/busy 데이터 수집
- 직접 busy/free 시간 입력
- 공통 가능 시간 계산 엔진: 15/30분 단위, 교집합
- 오프라인 회의 시 30분 버퍼 적용 로직
- 타임테이블 시각화: HTML 테이블 또는 그리드
- 회의 후보 시간 추천
- 추출 결과 사용자 검증 페이지
- 회의 안내 메시지 초안 생성 및 복사
- AWS Lightsail 배포 및 기본 보안

- 캘린더 스크린샷 Vision 인식
- 회의 초대장 자동 발송: 외부 이메일/메신저 연동
- 외부 캘린더 자동 일정 등록
- 회원 로그인/회원 관리 시스템: 익명 닉네임 기반으로 시작
