# 통합본 1차 피드백

전체 일관성 점수는 **78/100** 정도입니다. 제품의 중심축은 잘 잡혀 있지만, 몇 군데에서 범위가 넓어지면서 살짝 흔들립니다.

## 1. 대상 사용자

- 위치: `1.5 대상 사용자`, 첫째 문단과 둘째 문단
- 위치: `2.1 타깃 사용자 페르소나`, 첫째 페르소나와 둘째 페르소나

`1.5`에서는 1차 사용자를 “SW마에스트로 소마인”으로 정의하는데, `2.1`에서는 “IT 스타트업 PM”, “대학생 팀 프로젝트 조장”이 나옵니다. 제출용 문서라면 페르소나도 소마 맥락으로 맞추는 게 더 일관됩니다.

추천:

- 김PM → “소마 팀장/연수생”
- 박조장 → “소마 팀 프로젝트 조장”
- 대학생/스타트업 표현은 일반화 예시로 빼거나 삭제

## 2. 입력 방식과 MVP 범위

- 위치: `2.2 Agent의 역할`, “2. 개인 시간표 수집” 항목
- 위치: `3.2 핵심 기능 정의`, `[수집]`의 “3. 개인 시간표 입력 3종”
- 위치: `5.2 MVP 범위`, “MVP 범위는 다음과 같습니다” 아래 목록

앞부분에서는 Google Calendar 공유 링크, 스크린샷, `.ics`가 모두 핵심 기능처럼 보입니다. 그런데 `5.2`에서는 Google Calendar API는 MVP에 포함되고, 스크린샷 Vision은 제외됩니다. 범위가 조금 헷갈립니다.

추천:

- MVP: `.ics`, 직접 입력, 가능하면 Google Calendar API
- 확장: 스크린샷 Vision
- 문서 전체에서 이 우선순위를 반복해서 맞추기

## 3. 발송 기능

- 위치: `2.4 Agent의 자율성 범위`, “사용자 확인이 필요한 지점”의 마지막 항목
- 위치: `3.3 사용자 관점 워크플로우`, 주최자 관점 7번
- 위치: `3.4 시스템 관점 워크플로우`, `[사용자 의사결정 — 외부 액션 게이트]`
- 위치: `5.2 MVP 범위`, “이번에 포함하지 않는 것은 다음과 같습니다” 아래 두 번째 항목

문서에서는 “전송 버튼 클릭 후 안내 메시지 발송”이라고 되어 있는데, MVP 제외 목록에는 “회의 초대장 자동 발송”이 빠져 있습니다. 둘이 충돌하지 않게 표현을 나눠야 합니다.

추천:

- MVP에서는 “메시지 초안 생성 및 복사”
- 외부 이메일/슬랙/캘린더 실제 발송은 “이번에 제외”
- `전송 버튼` 대신 `복사 버튼` 또는 `초안 확인 버튼`으로 바꾸면 안전합니다.

## 4. 개인정보 보호와 Google API

- 위치: `1.6 핵심 가치`, 첫째 문단
- 위치: `4.4 데이터 활용 및 기억 관리`, 전체
- 위치: `5.2 MVP 범위`, Google Calendar API 항목

개인정보 보호를 강조하면서 Google Calendar API를 MVP에 넣으려면, 어떤 권한만 요청하는지 설명이 필요합니다. 지금은 “일정 제목, 장소, 설명은 저장하지 않는다”는 원칙은 있는데, Google API에서 free/busy만 쓰는지, 토큰을 저장하는지 여부가 부족합니다.

추천:

- “Google Calendar API는 free/busy 정보만 조회한다”
- “OAuth 토큰은 MVP에서 장기 저장하지 않는다” 또는 “서버 임시 저장소에 최소 기간만 저장한다”
- “이벤트 제목, 장소, 설명 권한은 요청하지 않는다”를 추가하면 설득력이 좋아집니다.

## 5. 기술 스택과 기능 범위

- 위치: `4.1 기술 스택`, 전체
- 위치: `5.2 MVP 범위`, Google Calendar API와 AWS Lightsail 항목
- 위치: `5.3 단계별 개발 로드맵`, 1주차 5/4~5/5와 2주차 5/13~5/14

기술 스택에는 Google API 관련 인증/OAuth 항목이 없는데, MVP와 로드맵에는 Google Calendar API 연동이 들어갑니다. 구현 계획과 기술 스택이 약간 덜 맞습니다.

추천:

- Google Calendar API를 MVP에 넣는다면 `4.1 기술 스택`에 `Google Calendar API`, `OAuth 2.0` 추가
- Google API를 확장으로 뺀다면 `5.2`, `5.3`에서 API 연동 테스트를 후순위로 이동

## 6. KPI

- 위치: `5.1 성공 지표(KPI)`, 첫 번째 항목

“마지막 주 생긴 방 수 / 처음 생긴 방 수”는 의미는 있지만, 현재 MVP 성공 기준으로는 조금 추상적입니다. 데모 제출용 문서라면 기능 검증형 KPI를 앞에 두는 게 더 좋습니다.

추천:

- 1순위: “5명 회의 기준 후보 시간 3개 추천 성공”
- 2순위: “일정 입력 정확도 90% 이상”
- 3순위: “회의 생성부터 후보 확인까지 5분 이내”
- “방 수 증가율”은 운영 지표로 뒤에 배치
