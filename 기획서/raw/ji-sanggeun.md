# 지상근 프로젝트 기획서

참여자: 지상근

## 1. 문제 정의 및 프로젝트 개요

### 프로젝트 한 줄 정의 [필수]

SW 마에스트로 연수생, 엑스퍼트, 멘토가 자신의 일정을 공개할지 직접 정하고, 사용 중인 캘린더에서 자동으로 불러와, 여러 사람의 공통 가능 시간을 빠르게 찾을 수 있도록 돕는 일정 조율 Agent 프로젝트

### 서비스 한 줄 정의 [필수]

사용 중인 캘린더와 소마 멘토링, 특강 일정을 자동으로 불러오거나 가능한 시간을 직접 입력하면, Agent가 모두가 모일 수 있는 시간 후보를 여러 개 찾아 추천해 주는 일정 조율 서비스

### 서비스 선정 배경 [권장]

기존 공유 캘린더로 일정을 잡으려면 회의마다 새 캘린더를 만들고 참여자를 한 명씩 초대하는 절차가 반복됨. 누군가는 일정 후보를 받아 직접 정리해야 하고, 그 정리 시간이 회의 자체보다 더 들 때도 있음.

또한 기존 도구는 일정 내용을 그대로 노출하거나 아예 비공개로 두는 양극단만 있고, "조율에 필요한 만큼만 공개"하는 중간 옵션이 부족함. 멘토 특강, 자유 멘토링 같은 SW마에스트로 도메인 일정도 별도 시스템에 있어 개인 캘린더와 따로 관리해야 함.

회의 셋업 자동화, 공개 범위 통제권, 도메인 일정 통합 — 이 세 가지를 한 번에 해결할 수 있는 도구가 마땅치 않아 직접 만들어 보기로 결정.

### 해결하려는 문제 [필수]

- 메신저로 일정을 조율하면 응답이 시간차로 흩어지고, 참여자가 늘어날수록 재확인과 정리 부담이 커짐. 일정 조율은 본질 작업이 아닌데도 팀의 집중 시간을 계속 소모.
- 공유 캘린더는 제목, 장소, 내용까지 노출되고 비공개는 조율에 필요한 정보가 부족함. "어디까지 공개할지"를 사용자가 직접 정할 수 있는 중간 옵션이 없음.
- 사람마다 Google, Apple, 네이버 등 서로 다른 캘린더 도구를 사용해, 일정을 한 형식으로 모으려면 수동 작업 필요.
- 멘토 특강, 자유 멘토링처럼 SW마에스트로 안에서 발생하는 도메인 일정은 별도 시스템에 있어, 사용자가 개인 캘린더에 별개로 저장해 둬야 함. 그렇지 않으면 회의 조율 시 이 일정을 빠뜨리거나 따로 확인 필요.
- 멘토는 정기 멘토링, 자유 멘토링, 특강을 동시에 운영하는 다중 관계 사용자라, 한 시간대에 여러 팀 요청이 겹쳐 더블 부킹 위험이 큼.
- 엑스퍼트는 본업과 후배 팀 상담을 병행하는데, 본업 캘린더와 상담 일정이 분리되지 않아 일정 충돌과 의도치 않은 노출 부담을 동시에 떠안음.

### 대상 사용자 [필수]

SW마에스트로 연수생, 엑스퍼트, 멘토 모두가 사용자
팀 회의, 멘토링, 엑스퍼트 상담, 코칭처럼 여러 사람이 시간을 맞춰야 하는 모든 상황에서 사용할 수 있으며, 주최자, 참여자 구분 없이 누구든 자신의 일정 공개 여부를 직접 선택해 참여 결정

### 핵심 가치 [필수]

일정 조율에서 사람이 직접 해야 했던 일(캘린더 생성, 참여자 초대, 일정 취합, 후보 정리)을 Agent가 대신 처리하고, 사람은 의사결정(공개 범위 선택, 최종 회의 시간 확정)만 남겨두는 것.

## 2. 사용자 및 Agent 설계

### 타깃 사용자 페르소나 [권장]

연수생, 엑스퍼트, 멘토 모두 사용자이지만 사용 패턴과 부담의 종류가 다름.

- 연수생: 4-6명 팀 단위로 활동. 팀 회의 주 1~3회, 엑스퍼트 상담 월 1~3회, 멘토링은 자유롭게 신청한 만큼(주 0회~) 진행하는 빈번한 주최자. 
- 엑스퍼트: SW마에스트로 1~4기수 위 선배로, 본업과 후배 팀 상담을 병행. 여러 후배 팀에서 들어오는 월 단위 상담 요청에 응답하는 참여자 입장. 본업 캘린더와 상담 일정의 분리가 페인포인트.
- 멘토: 외부 시니어 전문가(스타트업 대표, 시니어 엔지니어 등). 정기 멘토링, 자유 멘토링, 특강을 동시에 운영하는 다중 관계 사용자. 한 시간대에 여러 팀 요청이 겹치는 일이 빈번.

### Agent의 역할 [필수]

Agent가 자동으로 처리하는 일과 사람에게 남기는 결정을 명확히 분리.

Agent가 자동으로 처리하는 일 (반복 작업)

- 흩어진 일정 수집: 외부 캘린더(Google, Apple, 네이버) 자동 연동, SW마에스트로 멘토 특강·자유 멘토링 신청 내역 가져오기, 사용자가 직접 입력한 가능 시간을 한 번에 통합
- 입력 정규화: 형식이 서로 다른 일정 데이터를 공통 availability 형태로 변환
- 공개 범위 적용: 사용자가 일정별로 설정한 공개 수준(내용까지 공개 / busy만 공개)에 따라 참여자에게 노출될 정보를 자동 필터링
- 공통 시간 계산: 모든 참여자의 busy block을 겹쳐 비어 있는 30분 단위 슬롯 도출
- 후보 추천: 단일 답이 아니라 복수 후보 시간을 산출하고 각 후보의 추천 이유 생성
- 셋업 메시지 작성: 회의 확정 시 참여자에게 보낼 안내 메시지 초안 자동 작성

사람에게 남기는 결정 (의사결정)

- 일정별 공개 범위 선택: 어떤 일정을 내용까지 공개하고, 어떤 일정을 busy 표시만 할지
- 최종 후보 선택: Agent가 제안한 여러 후보 중 어느 시간으로 확정할지
- 발송 승인: Agent가 작성한 메시지를 실제로 보낼지 여부

### Agent의 자율성 범위 [필수]

Agent는 일정 수집부터 공개 범위 적용, 공통 시간 계산, 후보 추천, 메시지 초안 작성까지 사람의 개입 없이 끝까지 자동 처리. 그러나 외부 시스템에 변경을 일으키는 행위(캘린더에 일정 등록, 메시지 실제 발송, 외부 회의 확정 통보)는 직접 수행하지 않으며, 반드시 사람의 승인 이후에만 실행.

일정의 기본 공개 수준은 "busy만 공개"로 시작. 그 이상(제목, 장소, 내용 등)이 노출되려면 사용자가 명시적으로 공개를 풀어주는 액션이 필요(privacy by default).

## 3. 핵심 기능 및 사용자 흐름

### 주요 사용자 시나리오 [필수]

페르소나별로 3가지 대표 시나리오를 가짐.

시나리오 1 — 연수생이 팀 회의 시간을 잡는 경우 (주최자)
연수생 A는 팀원 5명과 다음 주 1시간 회의 시간을 잡아야 함. SomaTime에 회의 조건(이름, 1시간, 다음 주 평일)을 입력하고 팀원을 초대. 각 팀원은 본인 캘린더를 연결하거나, SW마에스트로 멘토링·특강 신청 일정을 API로 자동 불러오거나, 가능한 시간을 직접 입력하는 방식 중 골라 일정을 등록. Agent가 모든 일정을 모아 비어 있는 30분 슬롯 중 적절한 후보 3개를 제안하면, A는 그중 하나를 선택하고 팀원에게 보낼 안내 메시지 초안을 받아 발송.

시나리오 2 — 엑스퍼트가 후배 팀 상담 요청에 응하는 경우 (참여자)
엑스퍼트 B는 이번 달 들어 후배 팀 두 곳에서 1시간 상담 요청을 받음. SomaTime에 본업 캘린더를 연결해 두면, 후배 팀이 회의를 만들 때 B의 가능 시간이 자동으로 반영됨. 본업 일정은 "busy만 공개" 기본값으로 두어 누구와의 미팅인지 등 내용은 노출되지 않음.

시나리오 3 — 멘토가 여러 팀의 동시 요청을 한 번에 처리하는 경우 (다중 관계)
멘토 C는 한 시간대에 여러 팀의 멘토링 요청이 몰리는 일이 잦음. SomaTime에 자신의 가능 시간을 한 번 등록해 두면 여러 팀의 회의 생성에서 동시에 활용됨. 한 후보 슬롯이 한 팀에 확정되면 다른 팀의 후보에서는 자동 제외돼 더블 부킹을 방지.

### 핵심 기능 정의 [필수]

핵심 가치(자동화 vs 의사결정 분리)에 맞춰 두 묶음으로 정리.

Agent 자동 처리 기능 (반복 작업)

1. 회의 조건 구조화 — 회의명, 소요 시간, 가능 날짜 범위, 참여자, 필수 참석자 조건 입력 → 구조화
2. 다중 입력 소스 통합 — 외부 캘린더(Google, Apple, 네이버) 연동, SW마에스트로 멘토 특강·자유 멘토링 신청 내역 자동 가져오기, 사용자 직접 입력을 한 형태로 합침
3. 일정별 공개 범위 적용 — "내용까지 공개 / busy만 공개" 설정에 따라 다른 참여자에게 노출되는 정보를 자동 필터링
4. 공통 가능 시간 계산 — 모든 참여자의 busy block을 겹쳐 30분 단위 빈 슬롯 도출
5. 후보 추천 — 단일이 아닌 복수(3개 안팎)의 후보 시간을 산출하고 각 후보의 추천 이유 생성
6. 안내 메시지 초안 작성 — 회의 확정 / 재조율 시 보낼 메시지 초안 자동 작성

사용자 의사결정 기능 (사람이 직접)

1. 일정별 공개 범위 선택 — 일정마다 공개 수준 토글
2. 최종 후보 선택 — 추천된 여러 후보 중 확정
3. 발송 승인 — 메시지 초안을 보낼지 / 수정해서 보낼지 결정

### 사용자 관점 워크플로우 [필수]

[주최자 흐름]

1. 회의 조건 입력 (회의명, 소요 시간, 가능 날짜 범위, 참여자 목록)
2. 참여자 초대 링크 생성 → 메신저로 공유
3. (참여자 응답 대기 — 아래 참여자 흐름 진행)
4. Agent가 후보 시간 산출 → 주최자에게 알림
5. 후보 중 하나 선택해 확정
6. Agent가 작성한 안내 메시지 초안 검토 → 발송

[참여자 흐름]

1. 초대 링크로 접속
2. 일정 입력 방법 선택 — (a) 외부 캘린더 연결, (b) SW마에스트로 도메인 일정 자동 가져오기, (c) 가능한 시간 직접 입력
3. 일정별 공개 범위 설정 (기본값 "busy만 공개")
4. 제출
5. 주최자가 시간을 확정하면 안내 메시지 수신

### 시스템 관점 워크플로우 [권장]

```
회의 생성 요청
  → 회의 조건 구조화
  → 참여자 응답 수집
      ├─ Calendar Source Adapter (Google / Apple / 네이버)
      ├─ SomaDomain Adapter (멘토 특강 / 자유 멘토링)
      └─ Direct Input Handler
  → Privacy Filter (공개 범위 적용)
  → Availability Normalizer
  → Scheduling Engine (30분 단위 공통 슬롯 계산)
  → Candidate Ranker (후보 N개 + 추천 이유)
  → Message Generator (확정 / 재조율 메시지 초안)
  → 사용자 승인 대기
```

## 4. 기술 구현 설계

### 기술 스택 [필수]

- Frontend / Demo UI: Streamlit (빠른 프로토타이핑)
- Backend API: FastAPI
- Agent Workflow: LangGraph (자동 처리 단계와 사용자 승인 대기 노드를 분기 가능)
- LLM: Claude / OpenAI API (메시지 초안 작성, 추천 이유 생성)
- Data validation: Pydantic
- 일정 입력 어댑터
  - Calendar Adapter: `icalendar` (.ics), Google / Apple / 네이버 캘린더 API mock, 직접 입력 핸들러
  - SomaDomain Adapter: SW마에스트로 멘토 특강 / 자유 멘토링 신청 API mock
- Scheduling engine: Python deterministic time-slot calculation
- Storage: SQLite (로컬 데모용). 일정은 normalized availability 테이블만 저장하고, 제목·내용은 공개 설정이 풀린 항목만 별도 컬럼에 저장

### 시스템 아키텍처 [권장]

```
[입력 소스 레이어]
  ├─ Calendar Adapter (.ics / Google / Apple / 네이버)
  ├─ SomaDomain Adapter (멘토 특강 / 자유 멘토링)
  └─ Direct Input Handler
        ↓
[자동 처리 레이어 — Agent]
  Privacy Filter
    → Availability Normalizer
    → Scheduling Engine (30분 단위 공통 슬롯)
    → Candidate Ranker (후보 N개 + 추천 이유)
    → Message Generator (메시지 초안)
        ↓
[의사결정 레이어 — 사람]
  공개 범위 토글 / 후보 선택 / 발송 승인
        ↓
[외부 액션 — 사람 승인 후에만 실행]
  메시지 발송, 캘린더 등록 등
```

### 프롬프트 설계 전략 [권장]

LLM은 두 군데에서만 사용. 결정론적 계산은 LLM 밖에서 처리.

1. 추천 이유 생성 — 후보 시간 N개 각각에 대해 왜 이 시간이 적절한지 한 줄 설명. 입력 = (회의 조건, 후보 시간, 참여자 가능 시간 분포), 출력 = 정해진 양식의 한 줄 텍스트
2. 안내 메시지 초안 작성 — 회의 확정 / 재조율 메시지 초안. 입력 = (회의명, 확정 시간, 참여자), 출력 = 사용자가 그대로 보낼 수 있는 메시지

원칙

- 시간 계산 자체는 LLM에 맡기지 않음 (deterministic engine 사용)
- 출력 양식을 Pydantic 모델로 강제
- 개인 일정의 제목·내용은 프롬프트에 포함하지 않음 (공개 설정 무관하게)

### 데이터 활용 및 기억 관리 [권장]

저장하는 것 (회의 조율에 필요한 최소한)

- normalized availability — 시작·종료 + busy/free, 모든 사용자 일정의 공통 형태
- meeting metadata — 회의명, 소요 시간, 가능 날짜 범위, 참여자 ID
- 공개 설정 — 사용자가 명시적으로 풀어준 일정의 제목·장소·내용

저장하지 않는 것

- 공개 설정이 풀리지 않은 일정의 제목, 장소, 설명
- 외부 캘린더 OAuth refresh token 외의 인증 정보

기억 관리

- Agent는 stateful 메모리를 쓰지 않음. 한 번의 회의 조율은 stateless하게 종료
- 사용자 선호(예: "오전을 선호함")는 명시적 설정 항목으로만 저장, LLM이 자동 학습하지 않음

### 제약사항 및 예외 처리 [필수]

기술적 제약 (이번 범위 외)

- 외부 캘린더 OAuth 실시간 연동은 데모 범위에서 제외 — 샘플 .ics 파일 import로 대체
- SW마에스트로 멘토링 API는 실제 시스템 연동이 아닌 mock JSON 데이터 사용
- 반복 일정 최적화, 다중 시간대 처리는 이번 범위 외 (KST 단일 시간대로 제한)
- 로그인 / 회원가입, 자동 메시지 발송, 자동 일정 등록, 결제, 모바일 앱 미지원

예외 처리

- 공통 가능 시간이 없을 때: 가장 적은 충돌의 후보를 표시하고 충돌 인원 명시
- 참여자 일부가 응답하지 않은 상태에서 후보 산출 요청: 응답한 사람만으로 산출하되 미응답자가 누구인지 표시
- 동일 후보 슬롯이 여러 회의에 동시 선택될 경우 (멘토 시나리오 3): 먼저 확정한 회의 우선, 다른 회의의 해당 후보는 자동 무효화
- 외부 캘린더 데이터 파싱 실패: 해당 사용자에게 직접 입력 fallback 안내

## 5. 성과 평가 및 실행 계획

### 성공 지표(KPI) [권장]

데모 단계에서는 정량 측정이 어려우니 "이 정도면 성공"의 기준을 단계별로 분리.

기본 성공 (반드시 달성)

- 3-6명 회의 시나리오에서 후보 시간 3개를 안정적으로 산출
- 캘린더 연결, 소마 도메인 일정, 직접 입력 — 세 입력 경로가 모두 동작하고 결과로 통합됨
- "busy만 공개" 기본값에서 일정 내용이 다른 참여자에게 노출되지 않음

확장 성공 (가능하면)

- 멘토 시나리오 3: 동일 시간대에 두 팀이 같은 멘토를 후보로 잡을 때 더블 부킹 방지
- 빈 시간이 없을 때 fallback (충돌 최소 후보) 동작

성공 기준 (정성)

- 메신저로 일일이 묻는 기존 흐름과 비교했을 때 사용자가 "이게 더 빠르네"라고 느낄 수 있는 수준의 시간 단축

### MVP 범위 [필수]

반드시 구현할 것

- 새 회의 생성 (회의명, 소요 시간, 가능 날짜 범위, 참여자 입력)
- 입력 경로 3가지 모두: 직접 입력 + .ics 샘플 import + 소마 도메인 일정 mock JSON
- 일정별 공개 범위 토글 ("내용까지 공개 / busy만 공개")
- 30분 단위 공통 가능 시간 계산
- 후보 시간 3개 추천 + 각 후보의 추천 이유 (LLM 생성)
- 확정 / 재조율 메시지 초안 (LLM 생성)
- 사람의 승인 게이트 — 후보 선택, 발송 승인을 명시적으로 거치는 흐름

이번에는 하지 않을 것

- 외부 캘린더 OAuth 실시간 연동 (.ics import로 대체)
- 실제 SW마에스트로 시스템 연동 (mock JSON으로 대체)
- 로그인 / 회원가입
- 자동 메시지 발송, 자동 일정 등록
- 반복 일정 최적화
- 다중 시간대 (KST 고정)
- 모바일 앱
