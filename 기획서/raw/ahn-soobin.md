# 안수빈 프로젝트 기획서

참여자:

## 1. 문제 정의 및 프로젝트 개요

### 프로젝트 한 줄 정의 [필수]

팀원들의 캘린더를 자동으로 분석해 모두가 가능한 회의 시간을 즉시 추천해 주는 AI Agent를 만든다.


### 서비스 한 줄 정의 [필수]

"5월 1일 가능한 팀 회의 시간 알려줘" 한마디면 팀 전원의 공통으로 빈 시간을 찾아주는 회의 시간 비서.

### 서비스 선정 배경 [권장]

팀 단위로 일하는 환경에서 가장 자주 발생하지만 가장 비효율적으로 처리되는 일이 회의 시간 조율이다. 캘린더를 일일이 비교하거나, 메신저로 "언제 가능하세요?"를 반복하거나, When2meet 같은 별도 도구에 다시 입력하는 과정이 반복된다. 이미 각자 캘린더에 일정이 들어 있는데도 그 데이터가 활용되지 못하는 점에 주목해, 캘린더 데이터를 직접 읽어 자동으로 가능한 시간을 찾아주는 Agent를 기획했다.

### 해결하려는 문제 [필수]

핵심 문제: 팀 회의 시간 조율에 드는 시간과 커뮤니케이션 비용이 과도하게 크다.
문제라고 본 이유 1 — 같은 정보(각자의 일정)가 캘린더에 이미 존재하는데, 사람이 다시 묻고 답하며 수동으로 취합하고 있다.
문제라고 본 이유 2 — 팀원이 4명만 되어도 가능한 시간 후보가 기하급수적으로 늘어나 사람이 한눈에 비교하기 어렵다.
문제라고 본 이유 3 — Google Calendar, Outlook, Apple Calendar 등 사용하는 캘린더가 팀원마다 달라 통합 조회가 어렵다.


### 대상 사용자 [필수]

3~10명 규모의 소규모 팀 또는 프로젝트 그룹 (스타트업 팀, 학교 팀 프로젝트, 사이드 프로젝트 팀, 동아리 등)
일정 관리에 캘린더 앱(Google Calendar, Outlook, Apple Calendar 중 최소 하나)을 실제로 사용하는 사용자
회의 일정 잡기를 직접 주도해야 하는 사람(팀 리더, PM, 프로젝트 매니저, 조장 등)이 1차 사용자


### 핵심 가치 [필수]

"여러 명에게 묻고 기다리는 시간"을 "한 번 입력하고 결과 받기"로 줄인다. 즉, 회의 시간 조율을 채팅 한 줄로 끝내는 것이 이 서비스의 가장 큰 이점이다.


## 2. 사용자 및 Agent 설계

### 타깃 사용자 페르소나 [권장]

페르소나 A — 김PM: 27세, IT 스타트업 PM. 매주 5~6명 규모 팀의 정기 회의를 주관한다. 슬랙으로 "언제 가능?"을 묻는 데 누적 시간을 많이 쓴다고 느끼며, Google Calendar를 메인으로 쓴다.
페르소나 B — 이팀장: 34세, 디자인팀 리드. 팀원이 Outlook과 Google Calendar를 섞어 쓰기 때문에 통합 조회가 늘 골치다. 외부 미팅이 잦아 회의 시간 후보를 빠르게 좁혀줄 도구가 필요하다.
페르소나 C — 박조장: 21세, 대학생 팀 프로젝트 조장. 조원 4명이 모두 다른 시간표와 알바 일정을 가지고 있어, 회의 시간을 잡는 데만 며칠이 걸린 경험이 있다.

### Agent의 역할 [필수]

팀원 각자가 사용하는 캘린더(Google Calendar, Outlook 등)에 OAuth로 연결해 일정 정보를 읽는다.
사용자(주로 팀 리더)가 자연어로 "5월 1일 가능한 팀 회의 시간 알려줘"와 같이 입력하면, 해당 날짜·기간의 팀원 일정을 모아 비교한다.
팀이 미리 설정한 조건(회의 가능 요일·시간대, 회의 길이, 최소 참석 인원 등)을 만족하는 "공통 가능 시간 후보"를 추천한다.
결과를 시간순으로 정리해 보여주고, 사용자가 후보를 선택하면 캘린더 초대 초안까지 만들어 사용자에게 전달한다(전송 자체는 사용자가 직접 확인 후 실행).


### Agent의 성격 및 톤앤매너 [권장]

말투: 비서처럼 단정하고 간결한 존댓말.
스타일: 결과를 길게 설명하기보다 "가능한 시간 3개를 표로 보여주는" 식의 정보 우선 응답.
불확실할 때: 모호한 요청에는 추측해서 실행하지 않고, 짧은 후속 질문으로 확인한다(예: "몇 분짜리 회의를 잡으시겠어요? 30분 / 60분").


### Agent의 자율성 범위 [필수]

스스로 수행: 캘린더 일정 조회, 시간 비교, 후보 시간 산출, 회의 초대장 초안 작성.
사용자 확인 후 수행: 회의 초대 발송, 캘린더에 일정 등록, 외부 알림 전송.
수행하지 않음: 팀원 캘린더에 임의 쓰기, 사용자가 명시하지 않은 제3자에게 일정 전달, 비밀번호·결제 정보 등 민감정보 처리.
이유: "읽기는 자동, 쓰기·전송은 확인 후"라는 원칙으로, 자동화의 효용은 최대화하면서 사고는 막는다.



## 3. 핵심 기능 및 사용자 흐름

### 주요 사용자 시나리오 [필수]

시나리오 1 — 단일 날짜 가용 시간 조회: 팀 리더가 "5월 1일 가능한 팀 회의 시간 알려줘"라고 입력하면, Agent가 팀원 4명의 캘린더를 조회해 그날 모두 비어 있는 시간 슬롯(예: 10:00-11:00, 14:00-15:30)을 표 형태로 응답한다.
시나리오 2 — 기간 + 조건 조회: "다음 주 안에 1시간짜리 전체 회의 가능한 시간 3개 추천해줘" 입력 시, 영업시간(09:00-18:00) 안에서 5명 모두 가능한 1시간 슬롯 상위 3개를 시간순으로 추천한다.
시나리오 3 — 일부만 참석해도 되는 회의: "디자인팀 3명 중 2명만 가능해도 되니 이번 주 30분 회의 잡아줘" 같은 요청에 대해 최소 참석 인원 조건을 반영해 후보를 제시한다.
시나리오 4 — 후보 선택 후 초대장 초안: 사용자가 추천된 후보 중 하나를 고르면 Agent가 회의 제목, 시간, 참석자가 채워진 캘린더 초대장 초안을 만들어 보여주고, 사용자가 "보내기"를 눌러야 실제 전송된다.


### 핵심 기능 정의 [필수]

팀/멤버 등록: 팀을 생성하고 팀원 이메일을 추가한다. 팀원은 초대 링크로 자신의 캘린더를 OAuth 연결한다.
캘린더 연동(읽기 전용): Google Calendar API, Microsoft Graph(Outlook) 등에서 free/busy 정보 조회.
자연어 요청 해석: "5월 1일", "다음 주", "1시간", "오전" 같은 표현을 날짜·기간·시간 길이로 변환.
공통 가능 시간 산출: 요청 조건과 팀의 기본 정책(업무시간, 점심시간 제외 등)을 반영한 free/busy 교집합 계산.
결과 응답: 후보 시간 목록을 표로 출력. 각 후보별 참석 가능 인원과 비가용자 표시.
회의 초대장 초안 생성: 선택한 후보에 대해 캘린더 초대 메시지를 만들어 사용자 확인 후 전송.
사용자 관점 워크플로우
1) 팀 리더가 서비스에 가입하고 팀을 만든다.
2) 팀원에게 초대 링크를 보내고, 각 팀원은 자신이 쓰는 캘린더를 OAuth로 연결한다.
3) 팀의 기본 회의 정책을 설정한다(업무시간, 회의 가능 요일, 점심시간 제외 여부 등).
4) 채팅창에 자연어로 "5월 1일 가능한 팀 회의 시간 알려줘"를 입력한다.
5) Agent가 후보 시간 목록을 표로 응답한다.
6) 사용자가 후보 중 하나를 선택하면 초대장 초안이 생성된다.
7) 사용자가 내용을 확인한 뒤 "전송"을 눌러 실제 회의 초대를 보낸다.
시스템 관점 워크플로우
입력 처리: 사용자 메시지를 LLM 기반 의도 분석기에 보내 {action, date_range, duration, min_attendees, team_id} 같은 구조화된 파라미터로 변환.
데이터 조회: 각 팀원의 access token으로 free/busy API 호출. 결과를 정규화된 시간 슬롯 배열로 변환.
계산: 요청 기간을 N분 단위 슬롯으로 분해 후 모든 팀원에 대해 비교, 조건을 만족하는 슬롯을 추출하고 점수화(이른 시간/짧은 대기시간 등).
응답 생성: 상위 K개 후보를 자연어와 표로 포맷팅해 LLM이 최종 메시지 작성.
쓰기 작업: 사용자가 선택·확인한 경우에만 캘린더 API의 events.insert 호출.

4. 기술 구현 설계
기술 스택
프론트엔드: Next.js + React (간단한 채팅 UI 및 팀/캘린더 설정 화면)
백엔드: Python FastAPI (Agent 로직, 캘린더 API 호출, 시간 계산)
LLM: Anthropic Claude API (자연어 의도 분석 및 응답 생성)
외부 API: Google Calendar API, Microsoft Graph API(Outlook), Apple Calendar는 ICS 임포트 우선
DB: PostgreSQL (팀, 멤버, 토큰, 정책 저장) + Redis (free/busy 캐시)
인증: OAuth 2.0 (Google, Microsoft)
실행 환경: 로컬 데모 우선. Docker Compose로 한 번에 띄울 수 있게 구성.
시스템 아키텍처
Client(Web Chat UI) → API Gateway(FastAPI) → Agent Orchestrator
Agent Orchestrator는 LLM(의도 분석/응답 생성), Calendar Adapter(Google/Outlook), Scheduler Engine(가용 시간 계산기)을 호출한다.
Calendar Adapter는 각 캘린더 SDK 차이를 흡수해 통일된 free/busy 인터페이스를 제공한다.
토큰과 사용자 정책은 PostgreSQL에 저장하고, free/busy 결과는 Redis에 단기 캐시(예: 10분)해 같은 질문 반복 시 외부 API 호출을 줄인다.
프롬프트 설계 전략
역할 고정: "너는 팀 회의 시간 조율 비서다. 사용자의 요청을 먼저 구조화된 JSON으로 변환한 뒤, 결과만 간결한 한국어로 답한다."
입력 통제: 모든 사용자 입력은 1차로 "의도 분석 프롬프트"를 통과시켜 {date_range, duration, min_attendees, team_id} 등의 스키마로 추출. 모호하면 후속 질문 1개만 한다.
출력 통제: 최종 답변은 "표 + 간단한 요약" 형식을 강제. 후보가 없을 때는 가까운 대안 일자 2개를 제시.
환각 방지: 시간 후보 자체는 LLM이 만들지 않고 Scheduler Engine이 계산한 결과만 LLM에 넘겨 포맷팅하게 한다.
데이터 활용 및 기억 관리
단기 메모리(세션): 대화 내 "이전 후보" 같은 지시어 해석용. 세션 종료 시 폐기.
장기 기억(팀 단위): 팀의 기본 회의 정책(업무시간, 점심시간, 선호 회의 길이)과 자주 쓰는 회의 제목 템플릿을 저장.
개인 정보 최소화: 이벤트 제목·내용은 저장하지 않고 free/busy 시간 범위만 사용. 캐시는 짧게 유지(10분 수준).
우선순위 관점: 초기 데모에서는 메모리보다 "정확한 가용 시간 계산"이 핵심이므로, 메모리는 팀 정책 저장 정도로만 가볍게 시작.
제약사항 및 예외 처리
이번 범위에서 제외: 다국어 지원(한국어만), 외부 캘린더 "쓰기" 자동화(초안만 생성하고 전송은 사용자 확인), 모바일 네이티브 앱.
실패 케이스 — 토큰 만료: 만료된 팀원이 있으면 그 인원만 결과에서 제외하고, 재인증 안내 링크를 함께 보여준다.
실패 케이스 — 가용 시간 0개: "요청한 조건으로는 가능한 시간이 없습니다"와 함께 "조건 완화 시 가능한 후보"를 1세트 추천한다.
실패 케이스 — API rate limit: 캐시 사용, 동일 요청 통합, 사용자에게는 "잠시 후 다시 시도" 메시지.
개인정보·권한: 캘린더는 free/busy 권한만 요청. 이벤트 본문 read 권한은 받지 않는다. OAuth 토큰은 암호화 저장.
5. 성과 평가 및 실행 계획
성공 지표(KPI)
기능 KPI: 팀 5명 기준, 자연어 입력에서 가용 시간 후보 응답까지 5초 이내.
정확도 KPI: 추천된 시간이 실제로 모든 팀원이 가능한 시간일 확률 95% 이상(테스트 시나리오 기준).
사용성 KPI: 신규 사용자가 "가입 → 팀원 캘린더 연결 → 첫 회의 시간 추천 받기"까지 5분 이내 완료.
간단 기준: "3명 이상 팀에서 사람이 직접 조율하는 것보다 빠르고 정확하면 성공".

공통 가용 시간 계산 + 표 형태 응답
로컬 환경에서 Docker Compose로 한 번에 실행 가능한 상태

이번 프로젝트의 큰 목표인 "로컬 환경에서 실행 가능한 Agentic Workflow 데모 프로젝트"라는 기준에 맞춰 범위를 좁혔다.

기대 효과
팀 단위에서 회의 시간 조율에 들어가는 누적 시간을 의미 있게 줄인다(예상: 회의 1건당 평균 10분 이상 절감).
"캘린더 데이터를 사람이 다시 묻고 답하는" 비효율을 자동화로 대체한다.
Agentic Workflow의 실용 사례(외부 API 통합 + 자연어 인터페이스 + 사용자 확인 기반의 안전한 실행)를 데모로 보여줄 수 있다.


핵심 가치와 MVP가 이미 명확하다면, 이 항목이 없어도 제품을 만드는 데 직접적인 문제는 없다.
