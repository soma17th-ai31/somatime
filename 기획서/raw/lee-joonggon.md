# SomaTime 프로젝트 기획서

참여자: 이중곤

## 1. 문제 정의 및 프로젝트 개요

### 프로젝트 한 줄 정의 [필수]

개인 일정의 세부 내용을 공개하지 않고도 여러 사람의 공통 가능 시간을 찾는 일정 조율 Agentic Workflow 프로젝트입니다.

### 서비스 한 줄 정의 [필수]

캘린더 파일을 추가하면 일정 세부 내용을 공개하지 않고도 Agent가 가능한 시간대를 비교해 회의 후보 시간을 추천해주는 서비스입니다.

### 서비스 선정 배경 [권장]

SW마에스트로에서는 팀 회의, 멘토링, 코칭처럼 여러 사람이 함께 시간을 맞춰야 하는 상황이 자주 발생합니다. 현재는 주로 메신저로 가능한 시간을 주고받으며 조율하는데, 참여자가 늘어날수록 응답이 흩어지고 다시 확인해야 하는 일이 많아집니다.

공유 캘린더를 사용하면 일정 확인은 쉬워지지만, 개인 일정의 제목, 장소, 세부 내용이 노출될 수 있어 부담이 있습니다. 따라서 일정의 세부 내용은 숨기면서도 회의 조율에 필요한 가능 시간만 비교할 수 있는 방식이 필요하다고 판단했습니다.

### 해결하려는 문제 [필수]

첫째, 메신저로 가능한 시간을 주고받는 방식은 응답이 여러 대화에 흩어져 정리하기 어렵습니다. 특히 참여자가 많아질수록 누가 언제 가능한지 사람이 직접 다시 확인해야 합니다.

둘째, 여러 사람의 가능한 시간대를 사람이 직접 비교하면 시간이 오래 걸리고 실수가 생기기 쉽습니다. 단순한 회의 하나를 잡기 위해 반복적인 확인과 재조율이 발생합니다.

셋째, 캘린더를 그대로 공유하면 개인 일정의 제목, 장소, 설명 같은 민감한 정보가 노출될 수 있습니다. 회의 조율에 필요한 것은 실제 일정 내용이 아니라 특정 시간대가 가능한지 여부이므로, 불필요한 개인정보를 제거한 방식이 필요합니다.

### 대상 사용자 [필수]

1차 대상 사용자는 SW마에스트로에서 팀 회의, 멘토링, 코칭 시간을 자주 조율해야 하는 연수생입니다.

이들은 여러 사람의 가능 시간을 모아야 하지만, 각자의 개인 일정 세부 내용까지 공유하기는 부담스러운 상황에 자주 놓입니다. 따라서 일정 내용이 아니라 가능 여부만 안전하게 공유하고 싶은 사용자를 대상으로 합니다.

### 핵심 가치 [필수]

개인 일정의 세부 내용을 숨기면서도 회의 가능한 시간을 빠르게 찾는 것입니다.

사용자는 캘린더 원본을 그대로 공유하지 않아도 되고, Agent는 일정의 제목, 장소, 설명이 아니라 busy/free 정보만 활용해 공통 가능 시간을 비교합니다. 이를 통해 개인정보 노출 부담을 줄이면서 메신저로 반복 확인하는 시간을 줄일 수 있습니다.

## 2. 사용자 및 Agent 설계

### 타깃 사용자 페르소나 [권장]

프로젝트 팀장 역할을 맡은 SW마에스트로 연수생은 3-6명의 팀원, 멘토, 코치와 회의 시간을 자주 맞춰야 합니다. 하지만 각자의 가능 시간을 메신저로 모으면 응답이 흩어지고, 여러 시간대를 직접 비교해야 해서 피로감을 느낍니다.

또한 팀원들의 개인 일정 제목이나 장소까지 알고 싶은 것은 아니기 때문에, 일정 세부 내용 없이 가능 여부만 확인할 수 있는 방식이 필요합니다.

### Agent의 역할 [필수]

Agent는 캘린더 파일을 입력받아 회의 조율에 필요한 정보만 추출하는 역할을 합니다. 일정 제목, 장소, 설명 같은 세부 정보는 사용하지 않고, 각 참여자의 busy/free 시간대만 공통 형식으로 정리합니다.

정리된 시간대를 바탕으로 회의 조건과 참여자 가능 시간을 비교하고, 공통으로 가능한 후보 시간을 계산합니다. 이후 후보 시간 3개를 추천하고, 왜 해당 시간이 적절한지 간단한 추천 이유를 함께 제시합니다.

### Agent의 성격 및 톤앤매너 [권장]

Agent는 간결하고 실무적인 일정 조율 비서처럼 동작합니다. 사용자가 빠르게 회의 가능 시간을 확인할 수 있도록 불필요한 대화를 줄이고, 입력된 일정에서 어떤 정보가 사용되고 어떤 정보가 제외되는지 명확하게 안내합니다.

추천 결과를 제시할 때도 장황한 설명보다 후보 시간, 추천 이유, 확인해야 할 조건을 짧고 분명하게 보여줍니다.

### Agent의 자율성 범위 [필수]

Agent는 캘린더 파일을 분석하고 공통 후보 시간을 추천하는 단계까지만 자율적으로 수행합니다. 추천 결과에는 후보 시간과 간단한 추천 이유를 포함합니다.

최종 회의 시간 선택은 사용자가 직접 결정합니다. Agent가 임의로 회의 시간을 확정하거나, 참여자에게 자동으로 알림을 보내거나, 외부 캘린더에 일정을 등록하지는 않습니다.

## 3. 핵심 기능 및 사용자 흐름

### 주요 사용자 시나리오 [필수]

프로젝트 팀장이 팀원 3명과 멘토 1명의 1시간 멘토링 시간을 잡아야 합니다. 기존 방식이라면 메신저로 각자의 가능 시간을 물어보고, 여러 답변을 직접 비교해 회의 후보 시간을 찾아야 합니다.

SomaTime에서는 각 참여자가 직접 자신의 이름과 캘린더 파일을 추가하거나 busy/free 시간대를 제공합니다. Agent는 일정 제목, 장소, 설명 같은 세부 정보는 제외하고 가능 여부만 비교해 공통으로 가능한 후보 시간 3개를 추천합니다. 팀장은 추천 결과를 보고 최종 회의 시간을 선택합니다.

### 핵심 기능 정의 [필수]

1. 회의 조건 입력
회의 길이, 날짜 범위, 참여자 수 같은 기본 조건을 입력합니다.

2. 캘린더 파일 업로드
각 참여자는 직접 자신의 이름과 캘린더 파일을 추가합니다. MVP에서는 샘플 `.ics` 또는 CSV 파일 입력을 우선 지원합니다.

3. 개인정보 필터링
일정 제목, 장소, 설명 같은 세부 정보는 사용하지 않고 제거합니다.

4. busy/free 정규화
각 참여자의 일정을 공통 시간 블록 형식으로 변환합니다.

5. 공통 가능 시간 계산
모든 필수 참여자가 가능한 시간대를 탐색합니다.

6. 후보 시간 추천
조건에 맞는 후보 시간 3개를 추천하고, 각 후보의 추천 이유를 함께 제공합니다.

### 사용자 관점 워크플로우 [필수]

1. 사용자가 새 회의 생성을 선택합니다.
2. 회의 제목, 회의 길이, 날짜 범위, 참여자 수를 입력합니다.
3. 각 참여자가 직접 자신의 이름과 캘린더 파일을 추가합니다.
4. Agent가 일정 세부 정보는 제외하고 busy/free 정보만 추출합니다.
5. Agent가 공통 가능 시간을 계산합니다.
6. 사용자는 추천 후보 시간 3개와 추천 이유를 확인합니다.
7. 사용자는 후보 중 하나를 선택해 회의 시간을 확정합니다.

### 시스템 관점 워크플로우 [권장]

시스템은 먼저 회의 생성자가 입력한 회의 조건을 받고, 이후 각 참여자가 직접 추가한 이름과 캘린더 파일을 받습니다. 이후 캘린더 파일에서 회의 조율에 필요한 시간 정보만 읽고, 일정 제목, 장소, 설명 같은 세부 정보는 제외합니다.

정리된 시간 정보는 참여자별 busy/free 시간 블록으로 변환됩니다. 계산 엔진은 회의 길이와 날짜 범위를 기준으로 모든 필수 참여자가 가능한 시간대를 찾습니다.

마지막으로 Agent는 계산된 후보 시간 중 적절한 3개를 선택하고, 각 후보를 추천한 이유를 함께 출력합니다.

## 4. 기술 구현 설계

### 기술 스택 [필수]

프론트엔드와 백엔드를 분리해 구현합니다.

- Frontend: React
- Backend: Python, FastAPI
- Deployment: Vercel 또는 Netlify for frontend, Render 또는 Railway for backend
- Data validation: Pydantic
- Calendar parsing: `icalendar` 또는 CSV parser
- Storage: SQLite 또는 서버 임시 저장소
- Scheduling engine: Python 기반 deterministic time-slot calculation module

프론트엔드는 회의 생성자가 회의 조건을 입력하는 화면, 각 참여자가 자신의 이름과 캘린더 파일을 추가하는 화면, 추천된 후보 시간을 확인하는 화면을 담당합니다.

백엔드는 캘린더 파일 파싱, 개인정보 필터링, busy/free 정규화, 공통 가능 시간 계산, 후보 시간 추천 로직을 담당합니다. MVP 단계에서는 복잡한 기능보다 배포된 웹 데모에서 안정적으로 동작하는 핵심 흐름을 우선합니다.

### 시스템 아키텍처 [권장]

회의 생성자는 React 기반 프론트엔드 화면에서 회의 조건을 입력합니다. 각 참여자는 같은 프론트엔드 화면에서 자신의 이름과 캘린더 파일을 추가하고, 프론트엔드는 입력값과 파일을 FastAPI 백엔드로 전달합니다.

백엔드는 캘린더 파일을 파싱한 뒤 개인정보 필터를 통해 일정 제목, 장소, 설명을 제거합니다. 이후 참여자별 busy/free 시간 블록을 생성하고, 회의 조건에 맞는 공통 가능 시간을 계산합니다.

계산 결과는 후보 시간 목록과 추천 이유 형태로 프론트엔드에 반환됩니다. 프론트엔드는 사용자가 후보 시간을 비교하고 최종 선택할 수 있도록 결과를 화면에 보여줍니다.

### 프롬프트 설계 전략 [권장]

Agent는 일정 추천 이유를 생성할 때 입력된 일정의 제목, 장소, 설명 같은 세부 내용을 언급하지 않습니다. 추천 근거는 busy/free 여부, 회의 길이, 날짜 범위, 필수 참석자 가능 여부처럼 회의 조율에 필요한 정보로 제한합니다.

후보 시간을 설명할 때는 “참여자 전원이 가능함”, “요청한 회의 길이를 만족함”, “입력한 날짜 범위 안에 있음”처럼 검증 가능한 이유만 사용합니다.

불확실한 입력이 있거나 일부 참여자의 일정 정보가 부족한 경우에는 확정적으로 말하지 않고, 사용자가 추가로 확인해야 할 조건을 함께 안내합니다.

### 데이터 활용 및 기억 관리 [권장]

원본 캘린더 파일은 장기 저장하지 않고, 처리 후 삭제하는 것을 원칙으로 합니다. 서비스가 활용하는 데이터는 회의 조율에 필요한 최소 정보로 제한합니다.

일정 제목, 장소, 설명은 저장하지 않고, 시작 시간, 종료 시간, busy/free 여부만 공통 형식으로 변환해 사용합니다.

MVP 단계에서는 사용자 계정이나 장기 메모리를 만들지 않습니다. 실행 중 필요한 데이터만 사용하며, 회의 추천이 끝난 뒤에는 원본 일정 파일 없이도 결과를 확인할 수 있는 수준의 최소 데이터만 유지합니다.

### 제약사항 및 예외 처리 [필수]

캘린더 파일 형식이 지원되지 않거나 파일 내용이 깨져 있는 경우 파싱에 실패할 수 있습니다. 이 경우 사용자가 직접 busy/free 시간대를 입력하거나 지원되는 샘플 형식으로 다시 업로드하도록 안내합니다.

참여자의 일정 정보가 부족하면 정확한 추천이 어렵습니다. 일부 참여자의 정보가 누락된 경우에는 해당 사실을 명시하고, 추천 결과를 확정값이 아니라 참고 후보로 표시합니다.

모든 참여자가 가능한 시간이 없을 경우에는 조건을 만족하는 후보가 없음을 안내합니다. 이때 회의 길이를 줄이거나 날짜 범위를 넓히는 방식으로 조건 완화를 제안합니다.

MVP에서는 외부 캘린더 OAuth 연동을 지원하지 않습니다. 또한 Agent가 자동으로 일정을 등록하거나 참여자에게 알림을 발송하지 않습니다.

반복 일정, 복잡한 시간대(timezone), 종일 일정 처리는 MVP 단계에서 단순화합니다. 우선은 배포된 웹 데모에서 안정적으로 동작하는 기본 일정 조율 흐름을 완성하는 데 집중합니다.

## 5. 성과 평가 및 실행 계획

### 성공 지표(KPI) [권장]

샘플 캘린더 파일 4개를 업로드했을 때, 배포된 웹 서비스에서 공통 후보 시간 3개를 정상적으로 추천할 수 있으면 성공으로 봅니다.

추천 결과 화면과 백엔드 처리 데이터에 일정 제목, 장소, 설명 같은 세부 정보가 노출되지 않아야 합니다.

회의 생성자와 각 참여자는 별도 개발 환경을 설정하지 않고도 웹 페이지에 접속해 회의 조건 입력, 자신의 이름과 캘린더 파일 추가, 후보 시간 확인까지 수행할 수 있어야 합니다.

공통 가능 시간이 없는 경우에는 후보가 없음을 명확히 안내하고, 회의 길이 축소나 날짜 범위 확장 같은 조건 완화 방향을 제안해야 합니다.

### MVP 범위 [필수]

반드시 구현할 것:

1. React 기반 프론트엔드 배포
2. FastAPI 기반 백엔드 배포
3. 회의 조건 입력
4. 각 참여자가 직접 자신의 이름과 샘플 `.ics` 또는 CSV 캘린더 파일 업로드
5. 일정 제목, 장소, 설명 제거를 통한 개인정보 필터링
6. 참여자별 busy/free 시간대 정규화
7. 공통 가능 시간 계산
8. 후보 시간 3개 추천
9. 추천 결과와 추천 이유를 화면에 표시

이번에는 하지 않을 것:

1. 로그인/회원가입
2. 외부 캘린더 OAuth 실시간 연동
3. 자동 일정 등록
4. 자동 알림 또는 메시지 발송
5. 복잡한 반복 일정 처리
6. 장기 사용자 데이터 저장

### 단계별 개발 로드맵 [필수]

1. 요구사항과 데이터 형식 정의
회의 조건, 참여자 정보, 캘린더 파일 입력 형식, busy/free 시간 블록 형식을 먼저 정의합니다.

2. 캘린더 파일 파싱 및 개인정보 필터링 구현
샘플 `.ics` 또는 CSV 파일에서 시작/종료 시간만 추출하고, 일정 제목, 장소, 설명은 사용하지 않도록 처리합니다.

3. busy/free 정규화와 공통 가능 시간 계산 구현
참여자별 일정을 공통 시간 블록으로 변환하고, 회의 길이와 날짜 범위를 기준으로 모든 참여자가 가능한 시간을 계산합니다.

4. FastAPI 백엔드 API 구현
회의 조건과 캘린더 파일을 입력받고, 후보 시간 추천 결과를 반환하는 API를 구현합니다.

5. React 프론트엔드 구현
회의 조건 입력, 참여자별 이름과 캘린더 파일 추가, 추천 결과 확인 화면을 구현합니다.

6. 프론트엔드와 백엔드 연동
React 화면에서 FastAPI API를 호출해 실제 추천 결과가 표시되도록 연결합니다.

7. 배포 및 데모 시나리오 검증
프론트엔드와 백엔드를 각각 배포하고, 샘플 캘린더 파일 4개로 회의 후보 시간이 정상 추천되는지 확인합니다.

### 기대 효과 [권장]

회의 조율에 드는 반복적인 확인 시간을 줄일 수 있습니다. 사용자는 메신저에서 여러 사람의 가능 시간을 일일이 비교하지 않아도 되고, Agent가 공통 가능 시간을 계산해 후보를 제시하므로 더 빠르게 회의 시간을 정할 수 있습니다.

또한 개인 일정의 세부 내용을 공개하지 않아도 됩니다. 일정 제목, 장소, 설명은 제외하고 busy/free 정보만 활용하기 때문에, 회의 조율에 필요한 최소 정보만 공유하면서 개인정보 노출 부담을 줄일 수 있습니다.
