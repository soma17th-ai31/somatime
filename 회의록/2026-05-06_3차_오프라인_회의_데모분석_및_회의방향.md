## 2. 데모별 분석

### 2.1 SomaMeet_Joonggon

- 레포: https://github.com/soma17th-ai31/SomaMeet_Joonggon
- 스택: FastAPI, SQLite, React + Vite + TypeScript, shadcn/ui, Tailwind, Upstage `solar-pro3`, Docker Compose, Vercel 설정
- 주요 기능: 회의 생성, 닉네임 제출, 직접 busy/free 입력, `.ics` 업로드, LLM 후보 추천, 백엔드 후보 검증, 타임테이블, 확정 메시지

장점:

- 프론트엔드 완성도가 높다. 날짜 범위, 30분 단위 선택, 다크/라이트 테마, shadcn/ui 구성 등 실제 데모 화면으로 보여주기 좋다.
- LLM 추천 결과를 백엔드가 다시 검증한다. LLM이 시간을 지어내거나 조건 밖 후보를 내는 문제를 줄이는 구조가 좋다.
- `.ics`에서 시작/종료 시간만 저장하고 제목/장소/설명을 버리는 개인정보 원칙이 코드와 README에 명확하다.
- Docker Compose와 Vercel 배포 흐름이 있어 데모 배포 참고 자료로 쓸 수 있다.

단점/리스크:

- 결과 생성이 `UPSTAGE_API_KEY`에 강하게 의존한다. 키가 없으면 로컬 데모에서 결과 생성이 막힐 수 있으므로 deterministic fallback 모드가 더 필요하다.
- 백엔드는 단일 `main.py`에 API 흐름이 많이 들어 있어 장기 유지보수 구조는 다소 약하다.
- 테스트는 scheduling validator 중심의 소수 단위 테스트만 확인된다. 전체 API/프론트 흐름 검증은 부족하다.
- Google Calendar free/busy는 구현 범위에 없다.

통합 시 활용:

- 후보 검증 방식, 메시지 문구, polished UI 일부를 참고한다.
- LLM live 호출은 필수 경로가 아니라 optional로 둔다.

### 2.2 Somatime_Jiyeon

- 레포: https://github.com/soma17th-ai31/Somatime_Jiyeon
- 스택: FastAPI, SQLite, SQLAlchemy, React CDN + Babel 단일 페이지, 실행 스크립트 `run.sh`
- 주요 기능: 회의 생성, 닉네임/PIN, 직접 입력, `.ics`, Google free/busy adapter, 후보 랭킹, 메시지 초안

장점:

- `./run.sh` 하나로 백엔드와 정적 프론트엔드를 같이 띄우는 구조라 로컬 데모 진입 장벽이 낮다.
- API가 기획서 워크플로우와 1:1로 잘 매핑되어 있다. README와 코드 주석도 회의 단계 설명에 좋다.
- 닉네임 재진입용 PIN 해시가 있어 로그인 없는 환경에서 최소한의 본인 확인 아이디어를 제공한다.
- Google free/busy adapter를 분리해 두어 추후 OAuth 연동 구조를 설명하기 쉽다.
- 후보 랭킹이 deterministic 방식이라 API 키 없이 동작한다.

단점/리스크:

- 프론트엔드가 CDN + Babel + 단일 `app.js` 구조라 통합 제품의 유지보수 기반으로 쓰기에는 한계가 있다.
- Google Calendar는 실제 OAuth 완성이라기보다 adapter/interface 성격에 가깝다.
- 테스트 파일이 확인되지 않는다.
- 시간대 처리가 naive datetime 중심이라 KST/UTC 변환을 엄밀히 다루는 통합 코드에는 그대로 가져오기 어렵다.
- 예상 인원수, 참여자 확인 상태를 결과 생성 조건으로 강하게 묶는 장치가 약하다.

통합 시 활용:

- `run.sh` 같은 빠른 실행 경험, API 문서화 방식, PIN 아이디어, deterministic candidate ranker 설명을 참고한다.

### 2.3 Somameet_soobin

- 레포: https://github.com/soma17th-ai31/Somameet_soobin
- 스택: FastAPI, React + Vite
- 주요 기능: 회의 생성, 참여자 busy block 제출 API, in-memory 후보 계산, 간단한 결과 화면

장점:

- 코드가 매우 작아 일정 교집합과 fallback 후보 계산의 핵심 아이디어를 빠르게 이해할 수 있다.
- in-memory 구조라 알고리즘 스파이크나 설명용 샘플로 쓰기 좋다.
- 전원 가능 후보가 없을 때 가능한 인원이 많은 상위 슬롯을 보여주는 fallback 아이디어가 있다.

단점/리스크:

- 데이터가 메모리에만 저장되어 서버 재시작 시 모두 사라진다.
- 프론트엔드에는 참여자 busy block 입력 UI가 없어 README의 핵심 API를 직접 호출해야 한다.
- FastAPI CORS 설정이 없어 Vite 프론트에서 브라우저 요청이 막힐 가능성이 높다.
- `.ics` 파서, 메시지 초안, 권한 모델, 테스트, DB, 타임테이블 UI가 없다.
- 후보 검색 시간이 09:00-22:00으로 고정되어 있고 회의별 세부 시간대 설정이 없다.

통합 시 활용:

- 실제 코드 병합 대상보다는 알고리즘 sanity check와 교육/발표용 최소 예시로만 활용한다.

### 2.4 SomaMeet_Sanggeun

- 레포: https://github.com/soma17th-ai31/SomaMeet_Sanggeun
- 스택: FastAPI, SQLAlchemy 2.x, Alembic, SQLite, Pydantic v2, React + Vite + TypeScript, Tailwind, shadcn/ui, Playwright, pytest, LLM adapter(template/Gemini/OpenAI/Anthropic)
- 주요 기능: 회의 생성, 주최자/참여자 권한 분리, HttpOnly participant cookie, 직접 입력, `.ics`, Google OAuth URL, 타임테이블, 후보 계산, 메시지 초안, acceptance/unit/integration/e2e 테스트

장점:

- 구현 위임 스펙, API 계약, DB 스키마, 권한 모델, 테스트가 가장 체계적이다.
- 일정 계산 엔진이 LLM과 분리되어 있어 "계산은 deterministic, 설명은 Agent"라는 팀 원칙에 가장 잘 맞는다.
- 후보 0개일 때 "1명 빠진 후보" fallback, 그래도 없으면 조건 완화 제안까지 구현되어 있다.
- 주최자 토큰과 참여자 쿠키가 있어 확정 권한과 참여자 수정 권한을 분리한다.
- LLM 키가 없어도 template adapter로 메시지/추천 이유가 동작한다.
- README 기준으로 backend 70개 테스트, Playwright 골든패스, privacy guard가 준비되어 있다.
- 개인정보 보호가 DB 컬럼, ICS 파서, LLM prompt spy 테스트까지 이어져 있다.

단점/리스크:

- 5개 데모 중 규모가 가장 커서 오늘 바로 전체 구조를 팀원 모두가 이해하기에는 부담이 있다.
- Google OAuth와 LLM live path는 환경 변수와 외부 키 의존성이 있어 데모 당일 실패 가능성이 있다. template fallback을 기본 경로로 두어야 한다.
- 참여자 닉네임 재등록 시 기존 token을 다시 받을 수 있어, PIN 기반의 가벼운 본인 확인은 Jiyeon 데모보다 약하다.
- 프론트엔드가 기능 중심으로 잘 나뉘어 있지만, 최종 발표용 시각적 임팩트는 Joonggon/Sejong UI에서 일부 보강할 여지가 있다.

통합 시 활용:

- 통합 백엔드와 테스트 기준으로 채택한다.
- 프론트도 그대로 가져갈 수 있지만, 5/10 제출 전에는 UI 대수술보다 안정화와 README를 우선한다.

### 2.5 SomaMeet_Sejong

- 레포: https://github.com/soma17th-ai31/SomaMeet_Sejong
- 스택: Nx monorepo, pnpm, FastAPI, SQLAlchemy, PostgreSQL, Docker Compose, React 19, Vite, Tailwind, QR code, OpenAI message draft fallback
- 주요 기능: 회의 생성, 초대 URL/QR, host URL/token, 닉네임 참여, edit token, 직접 free/busy 그리드, ICS preview, 후보 계산, host dashboard, 메시지 초안

장점:

- monorepo와 `pnpm run setup`, `pnpm infra`, `pnpm dev` 흐름이 통합 개발 환경 관점에서 좋다.
- host/participant 화면이 분리되어 있어 주최자와 참여자 경험이 명확하다.
- QR 코드, host dashboard, 응답 현황 등 데모에서 보여주기 좋은 UI 요소가 있다.
- PostgreSQL + Docker Compose 구성이라 배포/운영 확장성을 설명하기 좋다.
- 참여자별 edit token으로 본인 availability 수정 권한을 분리한다.
- ICS preview가 이벤트 텍스트를 반환하지 않는 테스트가 있다.

단점/리스크:

- PostgreSQL, Nx, pnpm, uv를 함께 쓰므로 5/10까지 전원 개발 환경을 맞추는 비용이 생길 수 있다.
- Google Calendar OAuth는 제외되어 있다.
- 후보 계산은 전원 가능한 후보 중심이고, 1명 빠진 후보 같은 fallback 후보 추천은 약하다.
- 예상 참여 인원수보다 실제 응답자 수가 적어도 응답자 기준으로 후보가 산출될 수 있다.
- `either` 모드는 온라인/오프라인 확정 가능하다고 설명하지만, 오프라인 버퍼를 별도 후보로 검증하지 않는다.
- DB migration보다 `create_all` 중심이라 장기 운영 구조는 Sanggeun 데모보다 약하다.

통합 시 활용:

- QR, host dashboard, 링크 분리, monorepo 실행 스크립트 아이디어를 참고한다.
- 최종 베이스로 삼기보다는 UI/개발환경 아이디어를 부분 흡수하는 편이 안전하다.
